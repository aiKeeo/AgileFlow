# 入口意图识别与阶段路由

> **启用 Agileflow 时**，必须先执行本章，再读取对应 `phases/xx.md`。禁止跳过识别直接写代码或写文档。

## 何时启用 / 不启用

| 启用 ✅ | 不启用 ❌（直接回答或按 user rules 处理） |
|---------|---------------------------------------------|
| 用户 @agileflow 或说「走 agileflow / 走完整流程」 | 纯解释、答疑、概念问题 |
| 「从零做 XX 系统」「完整交付」类新功能 | code review、读代码、改一行 bug（走豁免） |
| 已有 `specs/` 且说「继续 agileflow」「下一步」（流程语境） | 未提流程的普通「写代码」「改 bug」 |
| 明确指定阶段：写 REQ、建模、出方案、验收测试 | 单文件小改、hotfix（走豁免，不进五阶段） |

**歧义时**：AskQuestion 确认是否启用 Agileflow，禁止默认套用全流程。

## 识别顺序（固定 4 步）

```
① 豁免判定 → ② 读取项目状态 → ③ 解析用户意图 → ④ 前置校验 & 确认 → 进入目标阶段
```

---

## ① 豁免判定（最先做）

命中下表 → 走豁免，**不进入五阶段**；未命中 → 继续 ②。

| 豁免类型 | 触发条件 | 执行方式 | AskQuestion |
|----------|----------|----------|-------------|
| **纯问答** | 仅解释，无代码/文档变更 | 直接回答 | ❌ |
| **微型改动** | 单文件 ≤20 行；无 API/DB/权限/支付变更 | 改代码 → **L1+L3** → 更新 todo | ❌ |
| **Hotfix** | 用户说 hotfix / 紧急修复 | **L1+L3**；核心路径 **L1–L3+L5 冒烟** | ❌ |
| **快速通道** | 用户说快速改 / 不走流程 | 微型改动 + 不写 REQ/002/003 | ❌ |

**豁免边界（任一命中走完整流程）**：API/DB/权限、支付/敏感数据、多模块、用户要求完整流程、跨 2+ 文件或 >20 行。

豁免只更新 `specs/todo.md`；不生成 REQ/002/003。L1–L5 见 [l1-l5-pipeline.md](../templates/l1-l5-pipeline.md)。

---

## 模式判定（快速 / 严谨）

| 快速 | 严谨 |
|------|------|
| 原型/Demo、简单 CRUD、开发 <1 天、无高并发 | 核心业务/服务端/DB、支付/权限/用户数据、开发 >1 天、团队协作 |

进入阶段后在首行声明中标注模式。

---

## ② 读取项目状态（判断「当前在哪」）

按顺序检查仓库与 `specs/`（不存在则视为**全新项目**）：

| 检查项 | 路径 | 状态含义 |
|--------|------|----------|
| 流程进度 | `specs/todo.md` →「流程进度」区 | 哪几阶段已 ✅ |
| 需求 | `specs/requirements/REQ-*.md` | 有文件但「草稿」→ 阶段 1 未完成；「已确认」→ 可进阶段 2 |
| 建模 | `specs/002-data-model.md` | 不存在或「草稿」→ 阶段 2；「已确认」→ 可进阶段 3 |
| 方案 | `specs/003-solution.md` | 不存在 → 阶段 3；存在且开发任务未清空 → 阶段 4 |
| 开发 | `specs/todo.md` →「开发任务」| 有未完成任务 → 阶段 4；全部 ✅ 且测试未 ✅ → 阶段 5 |
| 验收 | `specs/tests/README.md` | 已有 PASS → 交付已完成，问用户要维护还是新需求 |

**推导「建议阶段」** = 第一个未完成的前置阶段（流水线顺序 1→2→3→4→5）。

示例：
- 有已确认 REQ、无 002 → 建议阶段 **2**
- 有 003、开发任务剩 3 项 → 建议阶段 **4**
- todo 显示「开发实现 ✅、测试验收 ⬜」→ 建议阶段 **5**

---

## ③ 解析用户意图（判断「用户想干什么」）

### 关键词 → 目标阶段

| 用户表达（含但不限于） | 目标阶段 |
|------------------------|----------|
| 新功能、从零、完整流程、项目搭建、做一个 XX 系统 | **1**（或从建议阶段续跑） |
| 需求、澄清、PRD、用户故事、BDD、写 REQ | **1** |
| 建模、DDD、表结构、ER、聚合、sql、数据库设计 | **2** |
| 方案、架构、API、接口设计、技术选型、任务拆解 | **3** |
| 实现、开发、写代码、编码、补功能、按方案做 | **4** |
| 测试、验收、跑流水线、L1、L5、出验收报告 | **5** |
| 继续 agileflow、下一步（且存在 specs/）、接着做流程 | **建议阶段**（见 ②） |
| 只更新 todo、人类待办、标记完成 | **task-tracking** |

### 意图优先级

1. **用户明确指定阶段**（「帮我写需求」「先别开发只出方案」）→ 以用户为准
2. **用户说「继续 agileflow / 下一步」（且存在 specs/）** → 以 ② 建议阶段为准
3. **用户说新功能但未指定阶段** → 若 specs 为空从 **1** 开始；否则 **AskQuestion 确认**（见 ④）
4. **无法判断** → 必须 AskQuestion，禁止猜测

---

## ④ 前置校验 & 用户确认

### 硬规则：不可跳阶段（豁免除外）

| 目标阶段 | 最低前置条件 | 不满足时 |
|----------|--------------|----------|
| 2 建模 | ≥1 个 REQ「已确认」 | 退回阶段 1 或 AskQuestion |
| 3 方案 | 002「已确认」 | 退回阶段 2 或 AskQuestion |
| 4 开发 | 003 含 §测试环境 + §可观测性 | 退回阶段 3 |
| 5 测试 | 开发任务已完成（或用户明确只验部分） | 提示先开发或 AskQuestion |

用户**明确要求跳过**（如「快速通道 / 不要文档直接写」）→ 走 §① 豁免，不算跳阶段。

### 意图与状态不一致时

**必须 AskQuestion**，不可 silent 选阶段：

```
title: "Agileflow 阶段确认"
questions:
  - id: "target_phase"
    prompt: "检测到你希望【{用户意图}】，当前项目进度：{流程进度摘要}。\n请选择本次要从哪个阶段开始："
    options:
      - { id: "phase_1", label: "阶段 1：需求澄清" }
      - { id: "phase_2", label: "阶段 2：数据建模" }
      - { id: "phase_3", label: "阶段 3：技术方案" }
      - { id: "phase_4", label: "阶段 4：开发实现" }
      - { id: "phase_5", label: "阶段 5：测试验收" }
      - { id: "full_flow", label: "从阶段 1 走完整流程" }
      - { id: "resume", label: "从当前进度继续（建议：阶段 {N}）" }
```

- 仅当用户意图**唯一且前置已满足**时，可省略此卡片，但须在回复**首行声明**：
  `📍 Agileflow | 模式：{快速/严谨} | 阶段：{N}-{名称} | 依据：{用户原话或 todo 状态}`

### 全新项目 + 模糊需求

用户只说「做一个 XX」→ **阶段 1**，读取 [01-requirement.md](01-requirement.md)，先 AskQuestion 需求卡片（见 [templates/requirement-askquestion.md](../templates/requirement-askquestion.md)）。

## 路由决策表（速查）

| 场景 | 动作 |
|------|------|
| 纯解释、无交付物 | 豁免，不启用阶段 |
| 改一行 bug | 豁免 L1+L3 |
| 新项目 + 模糊想法 | 阶段 1 + 需求 AskQuestion |
| 有 todo、用户说「继续 agileflow」 | 读建议阶段 → 读对应 phase |
| 用户指定阶段且前置满足 | 直接进该 phase |
| 用户指定阶段但前置不满足 | AskQuestion：补前置 or 豁免 |
| 意图模糊 | AskQuestion 阶段确认卡片 |
| 仅维护 todo/humanTodo | [task-tracking.md](task-tracking.md) |

---

## 进入阶段后的行为

1. 输出阶段声明行（见上）
2. **只读一个** `phases/0N-*.md`
3. 该阶段完成后：若处于 1–4 → [askquestion-gate.md](../templates/askquestion-gate.md) 阶段闸门 → **停止**
4. 阶段 5 完成 → 更新 README，流程结束

## 正误示例

**✅ 用户**：「帮我把登录 API 实现了」
→ 读 todo/003 → 有 003 → 阶段 4；首行声明 → 读 04-development.md

**✅ 用户**：「继续 agileflow」
→ todo 显示数据建模 ✅、技术方案 ⬜ → 建议阶段 3 → 读 03-solution-design.md

**❌ 用户**：「写需求」但 REQ 已全部已确认、用户未说要改
→ 直接重写全部 REQ（应先声明状态并 AskQuestion 是修改 REQ 还是进阶段 2）

**❌ 用户**：「测试一下」但开发任务未做完
→ 直接跑 L5 标 PASS（应 AskQuestion 或提示先完成开发）
