# 人类待办与 BLOCKED-HUMAN

> task-tracking、阶段 5 testing 均引用本文件。

## 文件位置

- `specs/humanTodo.md`

## 文件模板

```markdown
# 人类待办事项

> 以下事项需要人类（产品、运维、业务方）完成，完成后请勾选并通知 AI 继续。

| 序号 | 事项 | 来源阶段 | 截止条件/说明 | 状态 |
|------|------|----------|----------------|------|
| 1 | 示例：提供微信支付商户号及 API 密钥 | 需求澄清 | 需联系财务 | ⬜ 待办 |

**状态标记**：
- ⬜ 待办（未开始）
- 🔄 进行中（人类已认领）
- ✅ 已完成（可继续后续流程）
```

## 状态与 AI 行为

| 标记 | 含义 | AI 行为 |
|------|------|---------|
| `⬜ 待办` | 未开始 | 阻塞，等待人类完成 |
| `🔄 进行中` | 人类已认领 | 可询问进度，继续等待 |
| `✅ 已完成` | 已完成 | 解除阻塞，继续流程 |

## 维护规则

| 操作 | 时机 |
|------|------|
| 创建文件 | 项目初始化时 |
| 追加事项 | 各阶段识别到外部依赖时 |
| 更新状态 | 用户反馈某项已完成时 |
| 阻塞检查 | **开发开始前**、**测试验收开始前** |

## 阻塞检查逻辑

1. 读取 `specs/humanTodo.md`，筛选 `⬜ 待办` 或 `🔄 进行中`
2. 检查项目根 `resource`、`resources/`、`.env.local` 是否已在应用中生效
3. **若存在阻塞项**：
   - 输出未完成序号列表，流程暂停
   - 开发开始前：终止开发或仅做不依赖外部资源的子任务
   - 测试验收开始前：仅允许 L1–L3 自动化子集；结论须为 `BLOCKED-HUMAN`
   - **硬阻塞**时禁止将 `specs/tests/README.md` 标为通过
4. **若无阻塞项**：继续执行

返回：`{ blocked: boolean, items: string[], allowAutomationOnly: boolean }`

## BLOCKED-HUMAN 结论

| 结论 | 含义 | 能否标「交付完成」 |
|------|------|-------------------|
| `PASS` | 0a 通过 + L1–L3 过 + 每个 REQ 验收报告达标 + L5 已执行或合规豁免 | ✅ |
| `BLOCKED-HUMAN` | humanTodo/资源未齐；仅跑了自动化子集 | ❌ |
| `FAIL` | 命令失败、场景 ❌、无取证 | ❌ |

**禁止**：humanTodo 仍有 ⬜ 时，将 L5 写 skip 却标「全流程交付完成」✅。

## 测试步骤 0（先于 L1）

### 0a 人类待办 / 外部资源

1. 读取 humanTodo，筛选阻塞级事项
2. 若阻塞 → 各 REQ 报告用 `BLOCKED-HUMAN`，暂停
3. **允许**：0a 未通过时仅 L1→L2→L3（mock），标注「自动化子集」

### 0b 项目资源文件

- 检查 `resource`、`resources/`、`.env.local`
- 未同步到配置 → 更新 humanTodo，未确认前不得标 L5 通过

### 0c 技术依赖

- 对照 `003 §测试环境与外部依赖`
- 仅当 0a 不阻塞或用户说「先跑 mock」时进入 L1
