# AskQuestion 规范

> 阶段 1–4 结束时**必须读本文件**并调用 **`AskQuestion` 工具**。

## 「停止」的准确含义（必读）

| 说法 | 含义 |
|------|------|
| **AskQuestion 后停止** | **本回复到此结束**，不在**同一条回复**里继续写下一阶段文档或写码 |
| **不是** | 永久不写文档；以后用户回答了也不写 |
| **用户回答/点选之后** | Agent **下一条回复必须**执行该阶段后续步骤（写 REQ、写 model、写 solution、写 dev…） |

**示例（阶段 1）**：
1. 回复 A：AskQuestion 需求卡片 → **停止**（不写 REQ）
2. 用户选完
3. 回复 B：**必须**写 `atlas/requirements/REQ-*.md` 草稿（禁止只说不写）
4. 回复 B 末：AskQuestion 确认草稿 → **停止**
5. 用户确认 → 标已确认 → AskQuestion 阶段闸门 → **停止**
6. 用户选「是，继续」→ 回复 C：**必须**读 `02-modeling.md` 或 `03-solution-design.md` 并**落盘**

## 阶段完成硬规则（最高优先级）

> **AskQuestion 只用于实质决策，不用于过程审批。**  
> 裁决冲突 → [SKILL 裁决表](../SKILL.md#裁决表冲突时以此为准)。

**阶段 0 / 1–4 产出完成后（二选一，然后必须停）：**

| 决策权 | 本回复末尾 | 下一阶段何时写 |
|--------|------------|----------------|
| **AI自主**（`AF_DECIDE=ai` 已确认） | **[审阅闸门](stage-delegation.md#审阅闸门ai-自主专属)** | 用户点「继续/不审继续」后的**下一条回复** |
| **user_decide** | **[阶段闸门](#阶段闸门模板)**（可与确认合并 1 卡） | 用户点「是，继续」后的**下一条回复** |
| **契约未确认**（无 env / pending） | **[流程启动卡](stage-delegation.md#流程启动卡首启强制)** | 用户答完后的**下一条回复**才进阶段 |

1. **禁止**同回复写下一阶段文档或写码（含 AI自主）
2. **禁止**用聊天文字代替卡片（「是否继续？」违规）
3. **无论决策权都必须问**：业务优先级 · 范围取舍 · 视觉偏好 · 需求/选型不确定性
4. **不可省**：F/MVP 阶段性确认卡、并行启动卡、同阶段首次方案确认（user_decide）
5. **可省（仅 `AF_DECIDE=ai` 已确认）**：阶段决策权卡、阶段内过程确认卡、可选构思审阅  
6. **不可省（首启）**：流程启动卡（模式+决策权），除非用户原话已同时点明

**阶段 5 完成后**不再 AskQuestion。

## 工具说明

Cursor 结构化提问工具名：**`AskQuestion`**。

| 场景 | AskQuestion |
|------|-------------|
| **首启 / 契约未确认** | ✅ **流程启动卡**（模式+决策权）→ [stage-delegation](stage-delegation.md#流程启动卡首启强制) |
| **阶段入口**（契约已确认） | `ai` → **跳过**决策权卡；`user` / 「我来」→ 问 → [stage-delegation](stage-delegation.md) |
| **阶段结束 · user_decide** | ✅ **阶段闸门** |
| **阶段结束 · AI自主** | ✅ **审阅闸门**（可 skip_review 继续，仍须先停本回复） |
| 阶段 1 澄清 / 草稿确认 | ✅ **仅 user_decide** |
| 阶段 3 技术栈 | ✅ **仅 user_decide** |
| 聊天式「是否继续」 | ❌ **禁止** |

## 通用规则

- 调用 AskQuestion 后 → **停止生成**，等用户点选
- ❌ 阶段结束不发审阅/阶段闸门就写下一阶段（AI自主也不行）
- ❌ AskQuestion 后同一回复继续产出
- ❌ 一个回复连续完成两个阶段（豁免除外）

## 阶段闸门模板

```
title: "开发流程确认"
questions:
  - id: "continue_next_stage"
    prompt: "{当前阶段名}已完成。是否继续进入【{下一阶段名}】阶段？"
    options:
      - id: "yes"
        label: "是，继续"
      - id: "no"
        label: "否，暂停"
      - id: "reselect_contract"
        label: "重选流程契约（快速/严谨 + 谁决策）"
```

| 选项 | 下条 |
|------|------|
| yes | 进下一阶段并落盘 |
| no | 暂停 |
| reselect_contract | → [契约重选](stage-delegation.md#契约重选旧项目--已确认后想改) |
| 当前阶段 | 下一阶段名 |
|----------|-----------|
| 项目盘点（init） | 需求澄清（或用户指定 sol/dev） |
| 需求澄清 | 数据建模（或按需→方案设计） |
| 数据建模 | 方案设计 |
| 方案设计 | 开发实现 |
| 开发实现 | 测试验收（tests:） |

**阶段 4 → 5 的闸门 prompt 须附**：可运行闸门 证据摘要（编译命令结果 + 冒烟结果）；无证据禁止发「是，继续」选项当已完成。

前缀单阶段模式：选项须含「暂停（本次仅完成本阶段）」——仍须 **AskQuestion**，不可静默结束。

## 阶段性确认卡（阶段 4 内 · MVP / F-xxx 切片强制）

> **编译+冒烟（可运行闸门）过了之后**，还要问用户要不要先看 / 继续。  
> **问用户 ≠ 替代可运行闸门**；C 没过禁止发本卡装完成。

### 何时必须 AskQuestion（命中任一）

1. 某个 **F-xxx** 相关 T 全部 ✅（且该模块可运行闸门 / 模块冒烟已过）  
2. 用户说「阶段性交付 / 先给用户看 / 演示 / MVP」→ **必须发本卡**（禁止 Agent 主观认定「还不能交」而跳过；可在 prompt 注明风险，但仍须问）  
3. 阶段 4 **全部** T ✅ → 用上面的**阶段闸门**（进 tests），本卡可并入或先发本卡再发阶段闸门  

> 切片以 **F-xxx / T-xxx** 为准（见 todo「功能依赖」）。勿另造 `MVP-n` 分组名；用户口头说 MVP 时走第 2 条。

### 「全部开发 / 连续做 / 连续做 / 不要停」不豁免

催进度、少发阶段内审阅卡 **不得**跳过本卡。  

**连续做可少的卡（白名单，仅此）**：可选「构思审阅」卡。  
**连续做不可少**：本阶段性确认卡、阶段结束闸门、可运行闸门、并行启动卡。

### 前端 Playwright 冒烟（可选增强 · 须询问 · **通用 FE**）

> 有任意可浏览器打开的前端时询问（Web / 后台 / 小程序-H5）。权威 → [fe-smoke-playwright.md](fe-smoke-playwright.md)。  
> **不绑框架**：正常启动 FE 后测；小程序只用 H5 形态，勿冒充 weapp 真机。

可与阶段性确认卡合并为 1 卡多题；**不可**不询问就声称「每页都测过」。

```yaml
  - id: fe_smoke_playwright
    prompt: |
      是否跑前端页面冒烟自动化？（Web/后台/小程序-H5 通用）
      须已启动前端；打开各页 + 收集 console/pageerror → atlas/logs/fe-smoke.*
    options:
      - id: run_fe_smoke
        label: "跑 Playwright 前端冒烟"
      - id: skip_fe_smoke
        label: "跳过（仅人工/构建冒烟）"
```

选「跑」→ Agent 须跑脚本并 Read report；FAIL 禁止说页面可展示。选「跳过」→ 证据写明用户跳过。

### 模板

```yaml
title: "阶段性可运行确认"
questions:
  - id: "milestone_runnable"
    prompt: |
      【{F-xxx}】相关任务已完成，可运行闸门已过：
      - 编译：{命令 + exit 0 / 失败摘要}
      - 冒烟：{主路径 + 结果}
      请确认下一步：
    options:
      - id: "continue"
        label: "通过，继续下一模块/任务"
      - id: "demo"
        label: "先给我看/演示（暂停开发）"
      - id: "fix"
        label: "有问题，先修再继续"
      - id: "pause"
        label: "暂停，稍后再说"
```

| 用户选 | Agent 必须 |
|--------|------------|
| 通过，继续 | **下条**继续下一 T / 下一 F（禁止只说不写） |
| 先给我看 | **停止开发**；给出启动步骤（端口/命令/账号）；等用户反馈 |
| 有问题 | 回阶段 4 修；修完再过可运行闸门 → 再发本卡 |
| 暂停 | 停止 |

**AI 自主**：仍须发本卡（可用审阅式措辞），**禁止**静默连做下一 MVP；选「不审继续」仅当选项里明确有等价「通过，继续」。

### 与可运行闸门 / 阶段闸门

| | 可运行闸门 | 阶段性确认卡 | 阶段 4 结束闸门 |
|--|--------|--------------|-----------------|
| 时机 | 每 T / 每模块冒烟时 | F/MVP 切片齐 + C 过 | 全部 T ✅ |
| 谁跑 | Agent 终端 | AskQuestion 用户 | AskQuestion 用户 |
| 可跳过？ | ❌ | ❌（全部开发也不豁免） | ❌ |

## 决策委派与闸门对照

| 决策权 | 阶段结束用什么卡 |
|--------|------------------|
| **用户决策** | [阶段闸门](#阶段闸门模板)（是否继续下一阶段） |
| **AI 自主** | [审阅闸门](stage-delegation.md#审阅闸门ai-自主专属)（确认 / **不审继续** / 调整 / 暂停） |

AI 自主时审阅闸门**替代**草稿确认 + 阶段闸门（一步完成确认与是否继续）。

## 模式差异（AskQuestion 次数）

> **与 [flow-modes](flow-modes.md) / [SKILL 裁决表](../SKILL.md#裁决表冲突时以此为准) 对齐**：快速默认可合并；严谨默认分步。用户显式「合并卡/少停」时可覆盖严谨。

| 阶段 | 快速（user_decide） | 严谨（user_decide） |
|------|------|------|
| 1 req | 需求卡 → 写 REQ → **确认+闸门 1 卡** | 需求卡 → 写 REQ → 确认卡 → 闸门卡（3 停） |
| 3 sol | 技术栈 → **落盘** → 确认+继续合并卡 | 技术栈 → 落盘 → 方案确认 → 闸门（禁与技术栈合并） |
| **AI 自主**（任一模式） | 跳过过程卡 → 落盘 → **审阅闸门 1 卡**（可 skip_review） | 同左 |

细则 → [flow-modes.md](flow-modes.md) · [stage-delegation.md](stage-delegation.md)

## 并行模式批次审阅（可选，默认不走）

仅用户显式并行时见 [parallel-orchestration.md](../phases/parallel-orchestration.md)。  
批次审阅**不能替代**阶段 4 全部完成后的**阶段闸门**。

## AskQuestion 不可用时的降级

1. 每张卡逐一输出（不可合并多卡为一条消息）
2. 编号格式：`1. {选项label}` / `2. {选项label}` …（与卡片 options 顺序一致）
3. 多题卡片：逐题输出，每题独立编号，用户须逐题回复
4. 用户回复非数字时：按语义最接近的选项匹配；无法匹配 → 重新输出选项并提示"请回复数字"
5. 首行标注：`⚠️ AskQuestion 降级模式（工具不可用）`
6. 输出编号选项后 **停止**，等用户回复

## 禁止行为

- ❌ **只 AskQuestion 不落盘**（用户已回答后仍不写 REQ/model/solution/dev）
- ❌ 用户选「是，继续」后只文字回复、不写下一阶段文件
- ❌ 阶段完成却不弹小卡片
- ❌ 自然语言代替 AskQuestion
- ❌ 未问技术栈就写 architecture.md（**user_decide** 时）
- ❌ humanTodo 阻塞时 tests 标 PASS
- ❌ **AI 自主但不落盘**
- ❌ 用户选「不审继续」后只说不写不写下一阶段
- ❌ F/MVP 切片做完不发**阶段性确认卡**就闷头下一模块
- ❌ 可运行闸门 未过就发阶段性确认卡 / 说「可以给用户看」
- ❌ 「全部开发」跳过阶段性确认卡
