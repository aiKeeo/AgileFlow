# AgileFlow 核心思想

> 一句话：把想法拆成可执行阶段，落到可运行交付——**流程本身就是产品**。

---

## 产品是什么

AgileFlow 要解决的不是「让 AI 多写点代码」，而是：

**把想法拆解到实现落地的一套流程。**

流程即产品。每个阶段有自己的规范：先把业务与设计说清楚，再构思，再写码，再验收。

防 AI 幻觉、不听指挥、假装完成，是后来加上的加固，**服从拆解，不反客为主**。

特别是：**用户把决策权交给 AI 时，摩擦可以少，证据不能少。**  
「你定 / 别问我」= 少澄清、少审阅卡；≠ 跳阶段、≠ 空跑勾选、≠ 先码后补。

---

## 主链

```
req → model(按需) → sol → dev(①②③) → test
```

（brownfield 另有前置 `init`：盘点已有代码与业务，不进主链叙事的「想法拆解」本体。）

「继续」= 下一条回复进下一阶段并落盘；禁止同回复跨阶段写下一阶段文档。

| 阶段 | 一句话 |
|------|--------|
| **req** | 把「要什么」写成可独立验收的需求 |
| **model** | 把「业务世界里有什么」说清楚——没有则几乎无视 |
| **sol** | 把「系统做什么、边界在哪、怎么暴露」定下来并拆成任务 |
| **dev** | 先盘可复用资源再构思；写法给人看得懂；再写码；再按 AC 证明做对了 |
| **test** | 证明整批可交，而不是单测绿就算完 |

---

## 各阶段核心思想

### req — 需求澄清

**解决什么**：用户到底要交付什么业务能力；怎样才算验收通过。

**核心思想**：

- 需求是**验收的唯一权威**。AC 用 Given/When/Then（即 BDD）写在 REQ 里；后面阶段只引用，不另造一套验收故事。
- **一功能一份 REQ**。MVP 是范围标签，不是把多个功能揉进一份「总览」交差的理由。
- 有界面时，结构与交互在 UID；视觉样式可以后定——先把「页面做什么」说清。**链了 UID 就必须有文件**，禁止空链标「已确认」。
- 信息已经够清楚就少问；没说清的只问缺口。问清楚是为了落盘，不是为了多聊几轮。

**本阶段结束时心里该有的东西**：已确认的 REQ（+ 按需 UID），人知道「做什么 / 怎么验收」。

**不要变成**：技术方案、接口明细、开发任务清单、「聊过但没落盘」，或 REQ 表里挂着 UID、目录却是空的。

---

### model — 数据建模（按需）

**解决什么**：业务实体、关系、规则、必要时的持久化形状——「世界里有什么、怎么连」。

**核心思想**：

- **看项目，不看仪式**。有复杂实体、状态机、关系变更 → 加重；简单 CRUD、无新实体 → **可几乎无视**。
- 「几乎无视」≠「假装没有这阶段」：跳过必须留下**建模判定**（依据写清），禁止静默进方案。
- 本阶段只回答业务世界是什么，**不**定义 API、不拆开发任务。
- 厚度跟模式走：快可以单文件；严谨可以五件套——都是在说清同一件事，不是为了凑文件数。

**本阶段结束时心里该有的东西**：要么有可用的 model；要么有白纸黑字的「跳过判定」。

**不要变成**：每个小改动都强制全量建模，或跳过成本与全量建模一样重。

---

### sol — 方案设计

**解决什么**：系统准备做哪些功能、边界在哪、对外暴露什么、技术怎么摆、开发按什么顺序做。

**核心思想**：

- 方案是**从需求到任务的桥**：功能（F）说清边界与暴露面；契约（contracts）承载接口/页面规格；架构全局一份；任务进 `todo`（每任务 ①②③）。
- **F 极简**：边界 + 暴露面；联调细节、字段映射、验收复述不放进 F——各有权威源，下游只链不抄。
- 技术栈要先有来源（人定 / AI 记 / 仓库已有），再写架构；未选型就写架构是空中楼阁。
- 任务编排默认串行；**快速模式的正当加速 = 并发作业**（并行启动卡），不是薄写 F/契约/todo。
- 「全部开发」不是默认开 Subagent 批量写码。
- todo 里的 ①②③ 此时应仍是未勾选——**清单是承诺形状，不是完成证明**。
- **快速与严谨的 sol 同质**：边界可对 AC、契约字段完整、开发打开就能动手——禁止「做登录相关」糊弄。

**本阶段结束时心里该有的东西**：可执行的功能边界 + 契约（按需）+ 架构 + 带档位的开发任务清单。

**不要变成**：把 REQ 再抄一遍；或在方案里提前写源码与测试报告；或提前把 ①②③ 勾满交差；或因「快速」把方案写成空壳。

---

### dev — 开发实现

**解决什么**：把方案里的每个任务真正做出来，并且跑得起来、对得上 AC；同时让人能看懂 AI 打算怎么做、做了什么。

**核心思想**：

- 每个任务固定三步：**① 构思 → ② 写码 → ③ 验 AC**。无合规构思，禁止写业务源码。
- **勾选 = 承诺已兑现**：勾① 必须已有 `atlas/dev/T-*.md`；勾③ 必须「## 结果」里有可运行证据；假标「开发实现 ✅」不算完成。
- **① 叙述五段式拆解（唯一质量线 = 完整）**：摘要 + 主流程 + 边界 + 实现说明 + 结果。逻辑块「怎么做」须编号 ≥2（含 if→错误码/回滚）；边界 ≥2 且挂「第 N 步」。接现有代码写清继续走 / 在…上加 / 照… / 新写——禁止无视库存造轮子。细则与 few-shot → `skills/agileflow/templates/dev-granularity.md`。
- **构思给人看，不是给脚本凑形**：摘要说清「做什么 / 不做什么 / 靠什么上游 / 验哪些 AC」；**做**里带接法。写在 `atlas/dev/`，业务源码写在工程正常位置。
- **可运行先于「写完」**：编译 / 能启 / 本任务冒烟——证据进构思的「结果」；单测绿 ≠ 可交；「待编译验证」却勾满 ✅ = 空跑。
- **没有精简/标准档**：厚度始终完整；**不跟任务个数走、不跟快速/严谨走**；AI 自主 / 快速模式都不减拆解。
- 默认一次推一个任务；**要快 → 并发**（快速模式推并行启动卡；严谨须人显式要）；每个任务仍先有合规①（落点可执行）。

**本阶段结束时心里该有的东西**：每个 T 有可读构思（流程清楚、落点清楚）、有代码、有可运行证据、AC 对得上。

**不要变成**：先码后补构思；一行「怎么做」糊弄；无视库存造轮子；因「你定/要快」把完整链路压成两句；用「checkbox 全绿」代替「真的跑过 / 人看得懂」。

---

### test — 测试验收

**解决什么**：整批功能能否交付；质量与回归是否站得住。

**核心思想**：

- 阶段 4 的 ③ 已经按 AC 写测并跑过；阶段 5 **不重写测试**，负责入场、归档、回归。
- **入场是 ③ 的延续**：同会话应复用已有可运行 / AC 证据，只补跨任务集成；跨会话或证据丢了才全量重验。
- 归档是把「做过」变成「可交接的验收记录」；回归是整仓再站岗。
- 单测绿、分层点名跑通，都不能单独冒充「全量可交」。

**本阶段结束时心里该有的东西**：验收报告齐、回归结论清楚（PASS / 阻断 / 失败），人知道能不能交。

**不要变成**：同会话对同一证据无意义地三重复跑；或缺测却硬标完成。

---

### init — 项目盘点（仅 brownfield，主链外）

**解决什么**：接手已有代码时，先知道业务与代码长什么样、写法锚点在哪。

**核心思想**：从零新建**不做** init；有存量才盘点。盘点是为了后续少幻觉、少撞命名，不是第四条主链阶段。

---

## 委托交付：用户不想自己决策时

> 常见原话：「你定」「别问我」「全程你看着办」。  
> 典型落盘：`AF_FLOW=fast` + `AF_DECIDE=ai`（未说严谨时默认如此；说了严谨 → `strict+ai`）。

| 可以少 | 不能少 |
|--------|--------|
| 阶段内澄清卡、技术栈追问 | 主链顺序、atlas 按阶段落盘 |
| `fast+ai` 阶段结束审阅卡（A 档绿 → 一行摘要） | todo ①②③、构思文件、可运行证据 |
| 用户逐步拍板 | 阶段结束**停**（下条「继续」才进阶） |

**易混**：

- 「快速」alone ≠ 不想决策（仍须问决策权）
- 「你定」≠ 跳阶段 / 快速通道豁免 / 先码后补
- AI 自主少问 → **令柱摩擦下降**，形柱与跑柱必须更硬，否则底层必偷懒

---

## 加固：形 / 令 / 跑（三柱对等）

设计中发现 AI 容易幻觉、不听指挥、空跑冒充完成，于是加了三层加固，**概念对等、执行有硬有软**：

| 柱 | 管什么 | 靠什么 | 约束力 |
|----|--------|--------|--------|
| **形** | 有没有按阶段写对；勾选是否对得上文件；假证据过不去 | `atlas/` 落盘 + `validate-atlas`（含 `TODO-CHECK-*`、`REQ-UID-断链` 等 A 档） | **硬阻断** |
| **令** | 听不听指挥；阶段能不能往下走 | 首启启动卡、阶段结束必须停、AskQuestion | 纪律（C 档） |
| **跑** | 是不是真能编译 / 启动 / 冒烟 | 可运行闸门；证据写入 `## 结果`；脚本不替你跑 | **硬阻断**（证据缺失不过） |

- **形**：拆解必须留下证据，才能续跑、审计、交接。勾了①却无构思、链了 UID 却无文件——形柱直接挡。
- **令**：关键决策与阶段切换由人指挥；用户说「你定」后，令柱停点变少，**不是取消形/跑**。
- **跑**：文档过闸 ≠ 交付；Agent 必须实际跑通。

三柱概念上对等。委托交付下令柱变软时，靠形柱/跑柱脚本硬挡补上——否则「全让 AI 干」就会空跑勾选、假装完成。

## 配套机制（短）

| 机制 | 作用 |
|------|------|
| **两维正交** | 快速/严谨 × AI自主/用户决策：只控停点与是否推并发；**不**减 sol/dev 质量；不用来跳主链、取消①或可运行闸门 |
| **委托交付（fast+ai）** | 用户不想自己决策的常见组合；少问仍须落盘与证据（见上节） |
| **质量线** | dev 唯一完整档：叙述五段式 + 逻辑块怎么做编号；fast/ai 不减厚度 |
| **豁免** | 问答/微型/hotfix 须首行声明；灰区 AskQuestion；快速模式 ≠ 快速通道 |
| **纠偏 L0–L3** | 中途发现不对先判级再动作；禁止硬扛继续写码 |

证据容器：`atlas/`（requirements / model / solution / dev / tests 物理隔离）。  
业务源码写在工程正常位置，不塞进 atlas。  
下游**只链不抄**：验收看 REQ AC，线框看 UID，接口看 contracts，边界看 F。

---

## 非目标

- 不是「纯防幻觉工具」——防失控服从拆解  
- 不做豁免场景的完整流水线（微型/hotfix/纯问答走豁免）  
- 脚本不替用户编译运行——只验落盘之形与勾选是否对得上证据  
- 不以任务数量决定流程重量  
- 严谨默认串行；快速可推并发，但「全部开发」≠ 自动启 Subagent  
- 不把「快速 / AI 自主」做成薄文档或跳证据的快速通道
