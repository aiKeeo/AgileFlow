# 快 = 并发，不是薄文档（对照例子）

> **一句话**：快速模式下 sol / dev **写法和严谨一样细**；时间省在「多 T 同时写码」，不是「少写几句」。  
> 规则权威 → [flow-modes](../templates/flow-modes.md) · [parallel 推并发](../phases/parallel-orchestration.md#快速模式推并发)

场景：登录（BE + FE 两个 T），用户说「快速，你定，要快点」。

---

## ❌ 错误「快速」——薄写换速度（禁止）

### 薄 F（开发看不懂边界）

```markdown
# [F-001] 登录
## 边界
**做**：登录相关能力
**不做**：其他
**暴露面**：有接口和页面
```

问题：AC 对不上；暴露面无编号；开发仍要猜「做多少」。

### 薄 dev（无法按步落码）

```markdown
# [T-001] 登录 BE
## 摘要
- 做登录接口
## 步骤
实现 Auth 相关逻辑，按需调整。
## 结果
待补齐
```

问题：无落点、无 AC、无接法；勾①/可运行都会假绿或过不了闸门。

### 时间线（看似快，实际返工）

```
薄 F → 薄 T-001 ① → 瞎写码 → 联调炸 → 补文档 → 再改码
```

---

## ✅ 正确「快速」——同质文档 + 并发作业

sol / dev **与严谨同档**；加速只发生在阶段 4 写码批。

### 同质 F（打开就能动手）

```markdown
# [F-001] 登录

- 关联 REQ：REQ-001
- ← REQ-001 · AC-001-01～06
- 优先级：P0
- **暴露面**：API-001, UI-001 → [contracts/](../contracts/)
- depends_on：—

## 说明

账号密码登录，签发登录态；小程序登录页提交后进首页。

## 边界

**做**：账号密码校验成功发登录态；失败统一文案；登录页提交与跳转（← AC-001-01～06）  
**不做**：注册、第三方登录、refresh（← REQ 范围外）  
**约定**：错误同文案；密码哈希存储
```

### 同质契约（字段写死）

```markdown
# [API-001] 登录
- 类型：HTTP POST · 路径：`/api/auth/login` · 鉴权：无

## 请求体
`{ "phone": "string", "password": "string" }`

## 响应
**200** `{ "token": "string", "user": { "id", "phone", "nickname" } }`

## 错误码
| HTTP | code | 说明 |
|------|------|------|
| 401 | 40101 | 账号或密码错误（统一文案，不区分不存在/错密） |
```

```markdown
# [UI-001] 登录页 · 路由：`/pages/login/index`
## 字段绑定
| 页面上 | 请求字段 | 接口 |
|--------|----------|------|
| 手机号框 | `phone` | API-001 POST body |
| 密码框 | `password` | API-001 POST body |
| 登录按钮 | — | 触发 POST API-001 → 存 token → 跳转首页 |
```

### 同质 todo（两端拆开）

```markdown
### T-001：[BE] 用户基础+登录 API — F-001 [标准]
- [ ] **① 构思落盘** → `atlas/dev/T-001-login-BE.md`
- [ ] **② 按 ## 步骤 写码**
- [ ] **③ 对照 REQ 验收 AC**

### T-002：[FE] 登录页 — F-001 [标准]
- [ ] **① 构思落盘** → `atlas/dev/T-002-login-FE.md`
- [ ] **② 按 ## 步骤 写码**（链 UI-001 §字段绑定）
- [ ] **③ 对照 REQ 验收 AC**
```

### 同质 dev · T-001（步骤有落点）

```markdown
# [T-001] 用户基础 + 登录 API — 构思 [BE]
- 档位：标准 · → F-001 · API-001

## 摘要
- **本 T**：F-001 后端；建 User 基础 + 登录 API；主责 AC-001-01～03。
- **做**：新写 `User`/`UserRepository`/`UserService.findByPhone`；新写 `AuthController.login` + `AuthService.validateAndSign`。
- **不做**：注册、refresh、小程序页。
- **上游**：F-001 · API-001 · depends_on 无。
- **AC**：AC-001-01、AC-001-02、AC-001-03。

## 步骤
| 步骤 | 动作 | 输入 → 输出 | 注意点（含落点） |
|------|------|-------------|------------------|
| **S1** | 建用户实体 | — → User 表 | **新写** `user.entity.ts` — phone 唯一、password_hash |
| **S2** | 按手机号查用户 | phone → User/null | **新写** `user.repository.ts` · `findByPhone` |
| **S3** | 登录入口 | JSON → 进入校验 | **新写** `auth.controller.ts` · `login` + `login.dto` |
| **S4** | 校验+签发 | password → token/401 | **新写** `auth.service.ts` · `validateAndSign`；错密与不存在同 40101 |

## 结果
| 项 | 证据 |
|----|------|
| 编译/启/冒烟 | `npm run build` ✅ · 正确→200+token · 错密→401 |
```

### 同质 dev · T-002（FE，可与 T-001 并发写码）

```markdown
# [T-002] 登录页 — 构思 [FE]
- 档位：标准 · → F-001 · UI-001 · depends_on：T-001（联调；写码可先 Mock）

## 摘要
- **本 T**：登录页提交 → 调 API-001 → 存 token → 进首页。
- **做**：照邻居页写法新写登录页；请求走已有 `services/api.ts`；字段绑定见 UI-001。
- **不做**：注册页、改鉴权拦截器。
- **上游**：UI-001 · UID-001 · API-001。
- **AC**：AC-001-04～06。

## 步骤
| 步骤 | 动作 | 输入 → 输出 | 注意点（含落点） |
|------|------|-------------|------------------|
| **S1** | 搭登录页结构 | — → 手机号/密码/按钮 | **新写** `pages/login/index` — 布局照 UID-001，不粘贴整图 |
| **S2** | 提交调登录 | 表单 → API-001 | 在 `services/auth.ts` 加 `login()` — body 字段对齐 UI-001 |
| **S3** | 成功跳转 | token → 首页 | 存 storage 后 `wx.switchTab` 首页；失败 toast 统一文案 |
```

---

## 时间线对比（效果长这样）

### 串行（严谨默认 / 未开并行）

```
主 Agent：写 T-001① → ② → 可运行 → ③
主 Agent：写 T-002① → ② → 可运行 → ③
总耗时 ≈ t(BE) + t(FE)
```

### 快速 + 并发（文档不减 · 时间重叠）

```
主 Agent：发并行启动卡 → 用户确认
主 Agent：批量写 T-001① + T-002①（两份都同质细）→ 各勾①
Subagent-A：T-001 ②→可运行→③
Subagent-B：T-002 ②→可运行→③   ← 与 A 同时
主 Agent：回收验收 → 勾 todo
总耗时 ≈ t(批量①) + max(t(BE写码), t(FE写码))
```

| | 错误快速 | 正确快速 |
|--|----------|----------|
| F / 契约 | 一两句糊弄 | 边界+字段齐全 |
| 每份 dev | 「实现逻辑」 | 流程表 + `` `Class.method` `` |
| 加速手段 | 少写字 | **≤3 T 并行写码** |
| 开发可读性 | 差，要再问 | 打开 atlas 就能干 |
| 闸门 | 常假绿/返工 | ①③证据齐 |

## Agent 自检（落盘前 10 秒）

1. 这份 F/dev 若删掉「快速」二字，是否仍够细？（不够 → 重写）  
2. 有没有用「相关 / 按需 / 待补齐 / 见代码」？（有 → 改成具体落点）  
3. 用户要快时，是否已推/已过并行启动卡，而不是在砍步骤？
