# AskQuestion 规范

> 阶段 1–4 结束时**必须读本文件**并调用 **`AskQuestion` 工具**。

## 「停止」的准确含义（必读）

| 说法 | 含义 |
|------|------|
| **AskQuestion 后停止** | **本回复到此结束**，不在**同一条回复**里继续写下一阶段文档或写码 |
| **不是** | 永久不写文档；以后用户回答了也不写 |
| **用户回答/点选之后** | Agent **下一条回复必须**执行该阶段后续步骤（写 REQ、写 model、写 solution、写 dev…） |

**示例（阶段 1）**：
1. 回复 A：AskQuestion 需求卡片 → **停止**（不写 REQ）
2. 用户选完
3. 回复 B：**必须**写 `atlas/requirements/REQ-*.md` 草稿（禁止只寒暄）
4. 回复 B 末：AskQuestion 确认草稿 → **停止**
5. 用户确认 → 标已确认 → AskQuestion 阶段闸门 → **停止**
6. 用户选「是，继续」→ 回复 C：**必须**读 `02-modeling.md` 或 `03-solution-design.md` 并**落盘**

## 阶段完成硬规则（最高优先级）

**阶段 0 / 1–4 的本阶段产出完成后：**

1. **必须**调用 `AskQuestion` 工具，弹出[阶段闸门](#阶段闸门模板)小卡片
2. **立即停止生成**——本回复不得再写下一阶段文档、不得写码、不得总结后继续
3. **禁止**用聊天文字代替小卡片（「是否继续？」「接下来做方案吗？」均违规）
4. **禁止**假设用户同意、禁止引用上一轮「是，继续」跳过**新阶段**的闸门

| 可省略（仅阶段**内** · **白名单**） | **不可省略** |
|------------------------|--------------|
| **连进/催进度唯一可少**：可选「构思审阅」卡 | **阶段结束闸门**（AI 自主→审阅闸门；用户决策→阶段闸门） |
| **快速模式**：阶段内确认可与闸门**合并为一张**（见 flow-modes；**须先落盘**；合并≠省略） | **阶段入口决策权卡**（仅 todo 未设全局 AI自主时） |
| **AI 自主**：阶段内需求卡/技术栈卡/草稿确认卡（用审阅闸门收口） | **AI 自主落盘 + 审阅闸门** |
| 并行批次 B 的可选构思审阅、批次间进度卡 | 严谨+用户决策：阶段 3 **首次**技术栈卡 |
| — | **F/MVP 阶段性确认卡**（连进/全部开发/「不要停」**均不豁免**） |
| — | **并行开闸卡**（说了并行也须确认） |
| — | 同阶段内**首次**方案确认（「不要停」**不得**跳过首次确认；仅禁止重复发同一张已确认过的卡） |

> **白名单外的卡一律不可省。** 「构思审阅（可选）」≠ F/MVP 阶段性确认卡。  
> 「不要停 / 连进」**不能**扩大白名单；可少卡仍只有「构思审阅」一类（见 [flow-modes 连进](flow-modes.md#连进模式快速专属)）。

**阶段 5 完成后**不再 AskQuestion。

## 工具说明

Cursor 结构化提问工具名：**`AskQuestion`**。

| 场景 | AskQuestion |
|------|-------------|
| **阶段入口** | ✅ 决策权卡（**仅未设全局 AI自主**）→ [stage-delegation](stage-delegation.md) |
| **阶段 0/1–4 结束（用户决策）** | ✅ **阶段闸门** |
| **阶段结束（AI 自主）** | ✅ **审阅闸门**（含跳过审阅继续） |
| 阶段 1 入口 / REQ 草稿确认 | ✅ **仅 user_decide 时** |
| 阶段 3 写 architecture 前技术栈 | ✅ **仅 user_decide 时** |
| 聊天式「是否继续」 | ❌ **禁止** |

## 通用规则

- 调用 AskQuestion 后 → **停止生成**，等用户点选
- ❌ 未 AskQuestion 就进入下一阶段
- ❌ AskQuestion 后同一回复继续产出
- ❌ 一个回复连续完成两个阶段（豁免除外）

## 阶段闸门模板

```
title: "开发流程确认"
questions:
  - id: "continue_next_stage"
    prompt: "{当前阶段名}已完成。是否继续进入【{下一阶段名}】阶段？"
    options:
      - id: "yes"
        label: "是，继续"
      - id: "no"
        label: "否，暂停"
```

| 当前阶段 | 下一阶段名 |
|----------|-----------|
| 项目盘点（init） | 需求澄清（或用户指定 sol/dev） |
| 需求澄清 | 数据建模（或按需→方案设计） |
| 数据建模 | 方案设计 |
| 方案设计 | 开发实现 |
| 开发实现 | 测试验收（tests:） |

**阶段 4 → 5 的闸门 prompt 须附**：闸门 C 证据摘要（编译命令结果 + 冒烟结果）；无证据禁止发「是，继续」选项当已完成。

前缀单阶段模式：选项须含「暂停（本次仅完成本阶段）」——仍须 **AskQuestion**，不可静默结束。

## 阶段性确认卡（阶段 4 内 · MVP / F-xxx 切片强制）

> **编译+冒烟（闸门 C）过了之后**，还要问用户要不要先看 / 继续。  
> **问用户 ≠ 替代闸门 C**；C 没过禁止发本卡装完成。

### 何时必须 AskQuestion（命中任一）

1. 某个 **F-xxx** 相关 T 全部 ✅（且该模块闸门 C / 模块冒烟已过）  
2. 某个 **MVP-n**（todo 分组）相关 T 全部 ✅  
3. 用户说「阶段性交付 / 先给用户看 / 演示 / MVP」→ **必须发本卡**（禁止 Agent 主观认定「还不能交」而跳过；可在 prompt 注明风险，但仍须问）  
4. 阶段 4 **全部** T ✅ → 用上面的**阶段闸门**（进 tests），本卡可并入或先发本卡再发阶段闸门

### 「全部开发 / 连续做 / 连进 / 不要停」不豁免

催进度、少发阶段内审阅卡 **不得**跳过本卡。  

**连进可少的卡（白名单，仅此）**：可选「构思审阅」卡。  
**连进不可少**：本阶段性确认卡、阶段结束闸门、闸门 C、并行开闸卡。

### 前端 Playwright 冒烟（可选增强 · 须询问 · **通用 FE**）

> 有任意可浏览器打开的前端时询问（Web / 后台 / 小程序-H5）。权威 → [fe-smoke-playwright.md](fe-smoke-playwright.md)。  
> **不绑框架**：正常启动 FE 后测；小程序只用 H5 形态，勿冒充 weapp 真机。

可与阶段性确认卡合并为 1 卡多题；**不可**不询问就声称「每页都测过」。

```yaml
  - id: fe_smoke_playwright
    prompt: |
      是否跑前端页面冒烟自动化？（Web/后台/小程序-H5 通用）
      须已启动前端；打开各页 + 收集 console/pageerror → atlas/logs/fe-smoke.*
    options:
      - id: run_fe_smoke
        label: "跑 Playwright 前端冒烟"
      - id: skip_fe_smoke
        label: "跳过（仅人工/构建冒烟）"
```

选「跑」→ Agent 须跑脚本并 Read report；FAIL 禁止说页面可展示。选「跳过」→ 证据写明用户跳过。

### 模板

```yaml
title: "阶段性可运行确认"
questions:
  - id: "milestone_runnable"
    prompt: |
      【{F-xxx 或 MVP-n}】相关任务已完成，闸门 C 已过：
      - 编译：{命令 + exit 0 / 失败摘要}
      - 冒烟：{主路径 + 结果}
      请确认下一步：
    options:
      - id: "continue"
        label: "通过，继续下一模块/任务"
      - id: "demo"
        label: "先给我看/演示（暂停开发）"
      - id: "fix"
        label: "有问题，先修再继续"
      - id: "pause"
        label: "暂停，稍后再说"
```

| 用户选 | Agent 必须 |
|--------|------------|
| 通过，继续 | **下条**继续下一 T / 下一 F（禁止只寒暄） |
| 先给我看 | **停止开发**；给出启动步骤（端口/命令/账号）；等用户反馈 |
| 有问题 | 回阶段 4 修；修完再过闸门 C → 再发本卡 |
| 暂停 | 停止 |

**AI 自主**：仍须发本卡（可用审阅式措辞），**禁止**静默连做下一 MVP；选「不审继续」仅当选项里明确有等价「通过，继续」。

### 与闸门 C / 阶段闸门

| | 闸门 C | 阶段性确认卡 | 阶段 4 结束闸门 |
|--|--------|--------------|-----------------|
| 时机 | 每 T / 每模块冒烟时 | F/MVP 切片齐 + C 过 | 全部 T ✅ |
| 谁跑 | Agent 终端 | AskQuestion 用户 | AskQuestion 用户 |
| 可跳过？ | ❌ | ❌（全部开发也不豁免） | ❌ |

## 决策委派与闸门对照

| 决策权 | 阶段结束用什么卡 |
|--------|------------------|
| **用户决策** | [阶段闸门](#阶段闸门模板)（是否继续下一阶段） |
| **AI 自主** | [审阅闸门](stage-delegation.md#审阅闸门ai-自主专属)（确认 / **不审继续** / 调整 / 暂停） |

AI 自主时审阅闸门**替代**草稿确认 + 阶段闸门（一步完成确认与是否继续）。

## 模式差异（AskQuestion 次数）

| 阶段 | 快速 | 严谨 |
|------|------|------|
| 1 req | 需求卡 → 写 REQ → **确认+闸门 1 卡** | 需求卡 → 写 REQ → 确认 → 闸门 |
| 3 sol | 技术栈 → **落盘** architecture+todo → 确认+继续合并卡 | 技术栈 → 落盘 → 方案确认 → 闸门 |
| **+ AI 自主** | 决策权卡 → 落盘 → **审阅闸门 1 卡**（可跳过审阅） | 同左 |

细则 → [flow-modes.md](flow-modes.md) · [stage-delegation.md](stage-delegation.md)

## 并行模式批次审阅（可选，默认不走）

仅用户显式并行时见 [parallel-orchestration.md](../phases/parallel-orchestration.md)。  
批次审阅**不能替代**阶段 4 全部完成后的**阶段闸门**。

## AskQuestion 不可用时的降级

输出编号选项（1. 是，继续 / 2. 否，暂停）→ **停止**，等用户回复数字。

## 禁止行为

- ❌ **只 AskQuestion 不落盘**（用户已回答后仍不写 REQ/model/solution/dev）
- ❌ 用户选「是，继续」后只文字回复、不写下一阶段文件
- ❌ 阶段完成却不弹小卡片
- ❌ 自然语言代替 AskQuestion
- ❌ 未问技术栈就写 architecture.md（**user_decide** 时）
- ❌ humanTodo 阻塞时 tests 标 PASS
- ❌ **AI 自主但不落盘**
- ❌ 用户选「不审继续」后只寒暄不写下一阶段
- ❌ F/MVP 切片做完不发**阶段性确认卡**就闷头下一模块
- ❌ 闸门 C 未过就发阶段性确认卡 / 说「可以给用户看」
- ❌ 「全部开发」跳过阶段性确认卡
