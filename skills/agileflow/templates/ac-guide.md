# AC（验收标准）与代码验收规范

> **阶段 1**：在 REQ 写 AC 表 — **AC 唯一定义处**，后续阶段只读、不改语义。  
> **颗粒度**：一条 AC ≈ **一条可独立红/绿的单测**（BE 默认 `1 AC ↔ 1 UT`）。  
> **阶段 4**：② 落地 `test/unit/`（主证据）→ 可运行闸门 → ③ 薄 `test/ac/`（出口烟测）+ 映射表。  
> **阶段 5**：归档 / 复跑（见 [05-testing.md](../phases/05-testing.md)）。  
> sol 从 REQ 提炼 F 边界 → [solution-core.md](solution-core.md)。

## 核心原则

| 关系 | 要求 |
|------|------|
| **AC 只在一处** | 定义在 REQ；**禁止**在 dev/方案/测试里新增或改写 AC 语义 |
| **AC ≈ 单测颗粒度** | 一条 AC = 一个 Given 组合 + 一次 When + 一族 Then；BE：**1 AC ↔ 1 个 UT 方法** |
| **UT 主证据 / ac 薄出口** | ② 写 `test/unit/`；③ `test/ac/` 只证明 HTTP/UI 出口，**禁止**称「AC单测」 |
| **1 AC ≥ 1 自动化证据** | 映射表须有路径；允许 `UT + 薄 ac` 同盖一条；禁止无映射勾 ③ |
| **观测面（REQ 填）** | `API` / `UI` / `规则` / `人工` — 只定从哪观察；测法在 sol/dev |
| **顺序** | ① → 勾① → ②（码+UT）→ **可运行闸门** → ③（薄 ac + 映射 + 回填） |
| **不可测 AC** | 观测面 `人工` 或状态 `⚠️ FE 人工`；不阻塞自动化 DoD |

## REQ：AC 颗粒度铁律

一条 AC = **一个角色 × 一种可独立观察的结果**（≈ 一个单测）。

**必须拆：**

1. Then 混了不同端（API 响应 vs 页面跳转）  
2. 用户/调用方可区分的不同失败  
3. 会单独回归的规则（如「不存在与错密同文案」）  

**不要拆：**

- 同一 Then 换无关夹具数据  
- 内部实现步骤（先查库再 compare）— 那是 UT arrange  
- 完整 JSON schema — 放 `contracts/`；Then 只写断言级结果  

**When**：业务动作或公开接口路径；**禁止**写内部类名 / 私有方法。  
**Then**：可断言（状态码/业务码/文案/是否发请求/是否跳转）；禁止「体验良好」。  
**条数**：单 REQ 常见 5–10；**>12** 先拆 REQ 或合并同质 AC。

### REQ AC 表

```markdown
| AC ID | 场景 | Given | When | Then（可断言） | 观测面 | AC 测试方法 | 状态 |
|-------|------|-------|------|----------------|--------|-------------|------|
| AC-001-01 | 登录成功·接口 | 用户存在且密码正确 | 提交登录 | 成功；token 非空 | API | （③ 后填） | ⬜ |
| AC-001-03 | 账号不存在同文案 | 账号不存在 | 提交登录 | 与 AC-001-02 同失败表现 | 规则 | （③ 后填 unit 路径） | ⬜ |
| AC-001-04 | 空提交 | 未填账号或密码 | 点登录 | 不发登录请求 | UI | （③ 后填） | ⬜ |
```

观测面：`API` | `UI` | `规则` | `人工`。

## sol：从 REQ 提炼功能与边界

| F 字段 | 提炼规则 |
|--------|----------|
| **做** | = 本 REQ 全部 AC 的 Then 并集（压成 1–3 句） |
| **不做** | = REQ「范围提示·范围外」∪ 故事邻域但无 AC 覆盖的能力 |
| **暴露面** | 由观测面汇总（有 API→API-xxx；有 UI→UI-xxx） |
| **回溯** | F 必须含 `← REQ-xxx · AC-xxx-01～nn`；**禁止**与 REQ 脱钩的边界注水 |

每条 AC 挂 **唯一主 T**（见 solution README / todo）。细则 → [solution-core.md](solution-core.md)。

## 阶段 4：② UT + ③ 薄验收

| 步骤 | 做什么 |
|------|--------|
| **①** | 一眼/摘要列出本 T 主责全部 AC ID + 观测面 |
| **②** | 业务码；**BE/规则/API 主责 AC → `test/unit/` 且 1 AC ↔ 1 方法**；UI 主责 → 前端单测/组件测同构 |
| **可运行闸门** | 编译 + 启/调 + 冒烟 |
| **③** | 对需证明出口的 AC 补薄 `test/ac/`（名含 AC ID）；填 **AC 映射表**；回填 REQ「AC 测试方法」 |

### test/unit（②）

- 路径以 `architecture.md` 为准（常见 `test/unit/` / `src/**/*.spec.ts`）  
- 方法名/描述须含或可映射到 AC ID  
- **主证据**：尤其观测面=`规则` 与全部 BE 主责 AC  

### test/ac（③ · 薄出口）

**可运行闸门之后**再写。推荐 `test/ac/req{NNN}_*`。

```
ac{REQ三位}_{序号}_{简述}
# 例：ac001_01_loginSuccess；描述含 AC-001-01
```

- ❌ 开发前写 `test/ac/`  
- ❌ 用 `test/ac` 代替 UT（BE 主责）  
- ❌ 称 `test/ac` 为「AC单测」  
- ❌ 未过可运行闸门勾 ③  

### ## 结果 · AC 映射表（勾 ③ 前必填）

| AC ID | unit | ac | 人工 |
|-------|------|-----|------|
| AC-001-01 | `test/unit/.../login_ok.*` | `test/ac/ac001_01_*` | — |
| AC-001-03 | `test/unit/.../unknownUser_same.*` | — | — |

## 阶段 4 DoD

| 步骤 | 动作 |
|------|------|
| **①** | 构思闸门过；本 T AC 列齐 |
| **②** | 步骤写完；BE 主责 AC 的 UT 已绿 |
| **可运行闸门** | 证据在 ## 结果 |
| **③** | 映射表齐；需出口的薄 ac 已绿；REQ 测试方法列已回填 |
| **演示前** | 存在端启动 + MVP 主路径冒烟 |

## 阶段 5（tests: / test:）

- 同会话且阶段 4③ 证据可解析 → **默认不复跑**，引用摘要更新 AC 状态；跨会话 / 缺证据 → 复跑  
- `test:unit` **不能**代替冒烟 / 可运行闸门 / 「可给用户看」  
- 全文「AC单测」一律改为：**单元测试** 或 **AC 验收测试**

## 禁止

- 开发前写 test/ac；dev 自造 AC；测试失败标完成  
- 多端糊进一条 Then；REQ 写类名/mock  
- 无映射勾 ③；用 unit 代替可运行闸门
