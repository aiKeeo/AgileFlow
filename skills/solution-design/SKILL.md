# 技术方案设计

## 执行流程

1. 基于已确认的需求和数据模型，设计系统架构
2. 定义 API 接口契约（RESTful 规范）
3. 拆分模块和文件结构
4. 拆解为原子任务（每个任务 ≤ 4 小时）
5. 等待用户确认

## 前置条件

- `specs/001-requirement.md`、`specs/002-data-model.md` 已确认

## 强制输出格式（specs/003-solution.md）

必须包含章节：

1. **技术栈**：后端、前端、数据库、其他依赖
2. **系统架构图**：ASCII 分层图
3. **文件结构**：目录树
4. **API 接口契约**：每个接口含方法、路径、请求体、响应体、错误码
5. **原子任务列表**：表格列——序号 | 任务名称 | 预计时间 | 依赖 | 输出文件

## 强制文档产出

| 文件 | 说明 |
|------|------|
| `specs/003-solution.md` | 技术方案全文 |
| `specs/todo.md` | 「技术方案」→ ✅ 已完成；将原子任务同步到「开发任务」区 |

## 同步 todo.md 示例

在 `specs/todo.md` 的「开发任务」追加：

```markdown
- [ ] 任务1：创建数据库表（30min）
- [ ] 任务2：实现订单实体类（20min）
```

## 强制规则

- 接口必须明确请求方法、路径、入参、出参和错误码
- 任务必须独立可执行、可测试
- 文件结构必须清晰，符合项目既有规范
