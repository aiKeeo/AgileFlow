# 产品需求澄清

## 执行流程

1. 提取用户的核心诉求、目标用户、业务价值
2. 对模糊需求进行苏格拉底式提问澄清
3. 输出标准 BDD 用户故事和验收标准
4. 明确范围边界（做什么 / 不做什么）
5. 生成需求规格书并等待用户确认

## 强制输出格式

必须按以下结构写入 `specs/001-requirement.md`（章节缺一不可）：

    # 需求规格说明书 v1.0

    ## 1. 核心信息
    - 项目名称：
    - 目标用户：
    - 业务价值：
    - 项目模式：快速模式/严谨模式
    - 创建时间：{{当前时间}}

    ## 2. 用户故事
    作为 [角色]
    我想 [功能]
    以便 [价值]

    ## 3. 验收标准（Given-When-Then）
    - 场景1：给定 [条件]，当 [操作]，则 [结果]
    - 场景2：给定 [条件]，当 [操作]，则 [结果]
    - 场景3：给定 [条件]，当 [操作]，则 [结果]

    ## 4. 范围边界
    ✅ 必须实现：
    ❌ 本次不做：

    ## 5. 约束与假设
    - 技术约束：
    - 业务假设：
    - 非功能要求：

## 强制文档产出

| 项 | 要求 |
|----|------|
| 路径 | `specs/001-requirement.md` |
| 内容 | 上述全部章节 |
| 代办 | 更新 `specs/todo.md`：「需求澄清」→ ✅ 已完成 |

## 强制规则

- 不明确的需求必须反问，禁止猜测
- 禁止输出任何代码，只输出规范文档
- 必须让用户签字确认后，才能进入下一环节
