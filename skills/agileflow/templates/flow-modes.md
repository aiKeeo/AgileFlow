# 流程模式：快速 vs 严谨

> **权威定义**：AskQuestion 次数、dev 段数、建模粒度均以本文件为准。  
> 首行声明须标注 `模式：快速` 或 `模式：严谨`。

## 铁律：快速 ≠ 跳阶段 ≠ 薄 todo（最高优先级）

> **哪怕是快速模式**，也必须按顺序走完整交付链；`atlas/todo` 仍然详细。  
> **「用户不用管」= 决策权 AI 自主**（少问选型），**不是**快速模式的含义。

```
需求澄清 → 数据建模（按需；跳过须写判定）→ 方案设计（含详细 todo）
  → 开发①构思落盘 → ②写码 → ③验收（含闸门 C）
```

| ❌ 快速模式禁止 | ✅ 快速模式允许 |
|----------------|----------------|
| 跳过 req/sol 直接写码 | 少问、合并确认卡（见下） |
| 跳过方案就进开发 | 建模按需跳过（须写「建模判定」） |
| 因「快速」把 todo 写成摘要/合并多 T/扁平 `- [ ] T-001` | **todo 与严谨同级**：`### T-xxx` + ①②③（含 dev 路径） |
| 把「快速」当成「用户不用管」 | 「用户不用管」→ 设 **AI 自主**（[stage-delegation](stage-delegation.md)） |
| 把「快速模式」当成豁免里的「快速通道」 | 快速通道仅微型改动；见 [00-intent-routing](../phases/00-intent-routing.md) |

**两维正交（勿混）**：

| 维度 | 控制什么 | **不**控制什么 |
|------|----------|----------------|
| **快速 / 严谨** | 文档厚度（dev 段、model 单文件）、AskQuestion **合并次数** | 阶段顺序、todo 是否详细、能否跳过①/闸门 C |
| **用户决策 / AI 自主** | 选型/澄清要不要问用户 | 阶段顺序、atlas 是否落盘、todo 详细度 |

## 模式判定（自上而下，先命中先定）

| 优先级 | 触发 | 模式 |
|--------|------|------|
| **1（强制·客观）** | `todo` 中 **T 头 ≥3**（行匹配 `^#{3,4} T-\d+`，仅 `###`/`####`）；或同时有 `[BE]`+`[FE]` | **严谨** |
| **1（强制·语义）** | 支付 / 权限 / **多模块** / DB·服务端 / 用户数据 / 开发 >1 天 / 团队协作 / 用户说「严谨」 | **严谨** |
| 2 | 单功能 CRUD、开发 <1 天、个人原型、用户说「快速 / Demo」，且**未命中 1** | **快速** |
| 3 | 无法判断 | AskQuestion 选模式（1 题） |

**硬覆盖**：「全部做 / 连续做 / 直接全实现 / MVP / 只看成品」**只表示催进度**，**不得**单独切快速，**也不得**当成并行许可或 Subagent 批量写码许可。  
命中优先级 1 → **必须严谨**；用户说「切快速」也 **拒绝**。

## 决策委派（与快速/严谨正交）

每阶段入口：**仅当** `todo` 决策委派未设全局 AI自主 时，问决策权（见 [stage-delegation.md](stage-delegation.md)）。全局已 AI自主 → **跳过**决策权卡，直接落盘 + 审阅闸门。

| 决策权 | 阶段内交互 | 阶段结束 |
|--------|------------|----------|
| **用户决策** | 按本文件下方规则（需求卡、技术栈等） | 阶段闸门 / 合并确认闸门 |
| **AI 自主** | **跳过**阶段内决策卡；Agent 推断并落盘 + `AI 决策记录` | **审阅闸门**（用户可不审直接继续） |

用户说「这阶段你定 / 后面都你定 / 不审了继续」→ 见 stage-delegation 话术表。  
**提醒**：AI 自主只少问用户；**阶段顺序与 todo 详细度不变**（见上方铁律）。

## 对比总表

| 维度 | 快速 | 严谨 |
|------|------|------|
| **阶段顺序** | **同左**：req →（按需）mod → sol → ①→②→闸门C→③；禁止跳阶段 | **同左** |
| **atlas/todo 详细度** | **同左**：流程进度 + 每 T 单独 ①②③；禁止摘要/合并多 T | **同左** |
| **阶段 1 AskQuestion** | 需求卡 → 写 REQ → **确认+闸门合并 1 卡** | 需求卡 → 写 REQ → 确认卡 → 闸门卡（3 停） |
| **阶段 2 建模** | 可单文件 `model-overview.md` 替代五件套 | 必须五件套齐全 |
| **阶段 3 AskQuestion** | 技术栈 → **落盘** architecture+**详细** todo → 确认+继续合并卡 | 技术栈 → 落盘 → 方案确认 → 闸门 |
| **阶段 4 dev** | 一/五/八/九；FE+UI 须三+`### 3.1/3.2/3.3`（**3.1 含线条图**）；二四六七可省略；五可执行 | **一～九全齐**；五可执行；FE 同左线条图要求 |
| **阶段 4 任务间** | 逐 T 完整 ①→②→闸门C→③（禁止批量写完所有 dev 再写码） | 同左 |
| **阶段闸门** | **不可省**（= 阶段闸门 **或** 审阅闸门，二选一） | 不可省 |
| **L4 覆盖率** | 出覆盖率报告即可（不卡阈值） | 变更模块 ≥80% |
| **首行声明** | 可简写：`📍 Agileflow 快速·阶段4·T-003` | 完整格式见 SKILL.md |

## 快速模式：AskQuestion 合并规则

### 阶段 1 — 确认+闸门合并

写完 REQ 草稿后，**一张卡**同时含：

```yaml
questions:
  - id: confirm_requirements
    prompt: "需求清单是否正确？（列出 REQ 摘要）"
    options: [确认 / 补充 / 大改]
  - id: continue_next
    prompt: "确认后是否继续下一阶段？"
    options: [是，继续（建模或方案） / 否，暂停]
```

- 选「确认」+「是，继续」→ 标 REQ 已确认 → **下条回复**进阶段 2/3 落盘  
- 选「否，暂停」→ 仅标已确认，不写下一阶段  
- **仍禁止**同回复写 model/solution

### 阶段 3 — 先落盘再确认（快速也适用）

**铁律：未写入 `architecture.md` + todo 前，禁止发「方案确认」。**

**快速路径**（仍须两停，可少一停）：

1. **AskQuestion 技术栈**（可单独；或与模式/决策权合并）→ 停  
2. **下条回复**写 features（若未写）+ `architecture.md` + `code-patterns` + `todo`  
3. **一张卡**：方案确认 + 是否继续（合并）→ 停  
4. 用户确认+继续 → **下条**进开发（本卡已含继续则**不再**单独阶段闸门）

**严谨路径**：技术栈卡 → 停 → 写 architecture+todo → 方案确认卡 → 停 → 阶段闸门 → 停。

**禁止**：合并卡里先「确认方案」再写 architecture（确认发生在落盘前 = 违规）。

## 快速模式：dev 构思精简（≠ 摘要/空壳）

> **精简** = 二/四/六/七 **可整段省略**（连标题也不写）。  
> **不是**只写「目标+八」；**不是**九个空 `##` 标题。

| 段 | 必填 |
|----|------|
| **一、五、八、九** | 标题+正文；五须过 [闸门 A4](../phases/04-development.md#a4-五可执行防空壳标题) |
| **三**（FE+UI） | 须有 `### 3.1` `### 3.2` `### 3.3` |

**严谨**：一～九标题全齐。勾①前过 [dev-quickstart 闸门 A](dev-quickstart.md)。  
**❌** 摘要版 / 空壳标题 / **假九段**（纯文本一、二、五无 `##`/`#### 5.`）/ 批量写多份 dev 再写码。

---

## 快速模式：建模精简

满足「须建模」时，可用 **单文件** 替代五件套：

```
atlas/model/
├── README.md
└── model-overview.md   # 含：实体列表、关系摘要、关键规则、表结构摘要
```

`README.md` 标已确认后等同五件套已确认。  
**严谨模式禁止**用单文件替代。

---

## 快速模式：REQ 入口例外

满足**全部**时可 **跳过第 1 步需求卡片**，直接写 REQ 草稿：

1. 用户 `req:` 前缀 **或** 明确「按此写 REQ / 不要问了」
2. 已含：目标用户、平台、≥1 核心功能、MVP 范围
3. 非变更已确认 REQ

仍须：写 REQ → **确认+闸门合并卡**（快速）或分步确认（严谨）。

## 「连续做」vs「连进模式」（勿混）

| 说法 | 谁可用 | 意思 | 不是 |
|------|--------|------|------|
| **连续做 / 全部做 / 直接全开发** | 快速 **与** 严谨 | 任务之间可连着做完整 ①→②→**C**→③，少停 | 跳过①；跳过闸门C；压缩模板；并行；Subagent 批量写码 |
| **连进模式** | **仅快速** | 在「连续做」之上，可少发阶段内审阅卡（见下节） | 严谨的「连续做」；跳过结束闸门 |

**严谨 +「直接全开发」** = 逐 T 完整一～九、可连续，**不适用**「连进模式」的少卡规则；结束闸门仍不可省。

---

## 连进模式（快速专属）

> **仅模式=快速时适用。** 严谨或命中强制严谨 → **不适用连进（少卡）**；但仍可「连续做」逐 T。见 [04-development](../phases/04-development.md#严谨-vs全部做--直接全开发)。

用户说「全部做 / 连续做 / 直接全开发」且模式=快速：

| 允许 | 禁止 |
|------|------|
| **同一大阶段内**连续落盘；少发**白名单**可选卡（仅「构思审阅」） | 跳过结束闸门；**跳过 F/MVP 阶段性确认卡** |
| **逐 T** 完整 ①→②→C→③（每 T 过闸门 A/B/C） | 跳过①；摘要/空壳；合并多 T；**批量写完所有 dev 再写码**；**跳过闸门 C** |
| — | **跨阶段连做**；先码后补洗 ✅；当成并行许可；**薄化 todo** |

**连进可少卡白名单（仅此 1 类）**：可选「构思审阅」。  
**绝不在白名单**：F/MVP 阶段性确认、阶段结束闸门、并行开闸卡、闸门 A/B/C。

**铁律不变**：大阶段结束 → 必须**结束闸门**（user_decide→阶段闸门；ai_decide→审阅闸门）→ 本回复停止。  
**连进 ≠ 跳阶段**：仍须 req→mod→sol→dev 按序；只是阶段内少卡、任务间可连续。

## 正误示例

**✅ 快速·阶段1**：用户长 PRD + `req:` → 直接写 REQ → 合并确认闸门卡 → 停  
**✅ 快速·阶段3**：技术栈卡 → 停 → 写 architecture+**详细** todo（每 T ①②③）→ 确认+继续合并卡 → 停  
**✅ 快速·dev**：逐 T 一/五/八/九（FE+3.x **含布局线条图**）；A0 → A → ① → B → ② → C → ③  
**✅ 快速 + AI 自主**：少问用户，但仍按序落盘 req→mod→sol→dev；todo 仍详细  
**✅ 严谨·「直接全开发」**：T≥3 强制严谨；逐 T 完整一～九（可连续，不可空壳/批量①）  
**✅ UID**：§2 区域表 + ASCII 布局线条图（缺图不可确认）

**❌ 快速就跳过需求/方案直接写码**  
**❌ 快速就把 todo 写成「实现 MVP」一条**  
**❌ 把「快速」当成「用户不用管」（应用 AI 自主）**  
**❌ T≥3 仍自称快速**  
**❌ 九个空 `##` 标题过闸门 A**  
**❌ 假九段：`一、…五、关键代码落点`（无 `##` / 无 `#### 5.`）**  
**❌ `## 一、目标` + `## 五、可执行方案` 未 A7 grep 仍勾①**  
**❌ FE 无 3.1 线条图仍勾①**
**❌ 先写完 T-002～011 的 dev 再统一写码**  
**❌ 摘要版 / 合并多 T / 未落盘就确认方案**  
**❌ UID/FE 布局只有文字表、无线条图**
