# 阶段 4：开发实现

> **执行清单（先读）**：[dev-quickstart.md](../templates/dev-quickstart.md)  
> 模板：[dev-rationale.md](../templates/dev-rationale.md) · 范例：exemplar-BE / exemplar-FE · 写法锚点：[code-conventions.md](../templates/code-conventions.md)

## 规则（构思先行）

| 规则 | 说明 |
|------|------|
| **无构思禁写码** | 无合规 dev → 禁止 Write 业务源码 |
| **构思须落盘** | 构思必须落盘；聊天不算 |
| **一任务一文件** | 1 T = 1 文件；禁止多 T 糅合 / 合并勾选 |
| **文档按风险分档** | **唯一厚度轴**：全档 **摘要+步骤+结果** 三段；标准+摘要五 bullet；完整=标准+字面量严检（见下） |
| **步骤可执行** | template 模式：每 `####` 须 **涉及改动** + 代码落点（`` `method` ``）；legacy（`AF_TEMPLATE=no`）：**用户/系统/改** |
| **②③分工** | ② 按 `## 步骤` 写码；③ 对照 REQ AC 写 test/ac 并跑绿 |
| **FE 链 UI 契约** | FE dev **链** `contracts/UI` §字段绑定；**禁止** dev 内字段映射表、禁止 F 联调卡 |
| **默认串行** | 一次只推进 1 个 T；禁止批量写完所有①再写码。**例外**：用户显式并行 **且** 已过 [parallel 并行启动卡](parallel-orchestration.md#并行启动卡强制) **且** 走批量构思；单批 **最多 3 个 T** 的① |
| **连续做≠压缩** | 连续做 ≠ 压缩模板 |
| **事后补写不勾①** | 事后补写/摘要/空壳 → 不勾 ①；父任务不得伪装成正式 ✅；**即使用户说「接受技术债」也禁止**把「开发实现」标 ✅（只允许标 `⚠️ 事后补写归档`，须另开正式①重做才能 ✅） |
| **闸门按档** | 精简/标准档 1 道写码前检查 / 完整档 2 道（构思+写码）含字面量 grep；有文件但 grep 不过 → **禁止勾①**（仅完整档） |
| **禁止伪并行** | 「全部开发」≠ 并行；**禁止**无显式并行就启 Task/Subagent 写业务源码；1 Task=1 T 且须已有合规① |
| **开发完成格式门槛** | 标「开发实现 ✅」前须**同时**满足：① 合法 T 头（`^#{3,4} T-\d+`，**仅 ###/####**）≥1；② 每个 T 头下有 ①②③ 三行（含 `atlas/dev/T-xxx` 路径）；③ `count(atlas/dev/T-*.md)`（非⚠️、非 README、非 temp/）= 合法 T 头数；④ 每 T 可运行闸门证据齐。扁平列表或有头无三段式 → **格式违规禁止 ✅** |
| **TodoWrite 强制展开** | 进阶段 4 /「全部开发」：**先 TodoWrite 展开**——每个 T **必须三条**「①构思 / ②写码 / ③AC」；禁止一条 Todo 多 T；未展开禁止 Write 业务源码（见 [todo.md](../templates/todo.md#todowrite-强制展开)） |
| **可运行闸门** | 每 T 勾③前必须 **编译 + 能启/能调接口 + 本 T 冒烟**；F-xxx/MVP 切片完再跑模块冒烟；「给用户看」前存在端启动冒烟全过；证据写入 `## 结果`；单测绿 ≠ 可交（见 [dev-quickstart 可运行闸门](../templates/dev-quickstart.md#可运行闸门每-t--每模块强制)） |
| **像素对比**（有强制原型） | 勾③前须 [fe-pixel-compare](../templates/fe-pixel-compare.md) PASS；闸门扫 report |

催进度反模式 → [SKILL.md](../SKILL.md)。Subagent 细则 → [parallel-orchestration.md](parallel-orchestration.md)。

## 风险分档

> 替代旧 T≥3 强制严谨规则。**流程重量与任务风险成正比，不与任务数量成正比。** 分档由风险维度决定，不由 T 的数量决定。

### 三档定义

| 档 | 必填段 | 闸门 | 适用 |
|----|--------|------|------|
| **精简** | `## 摘要` · `## 步骤` · `## 结果`（摘要含 **本 T**；步骤≥1） | 写码前 + 可运行 | 单端·无DB·无权限/支付·<100行·无跨模块 |
| **标准**（默认） | 摘要须 **本T/做/不做/上游/AC** 五 bullet（步骤≥2） | 同上 | 默认 |
| **完整** | **同标准** + 字面量严检（步骤≥3） | 构思+写码+可运行 | 支付·权限·DB·多模块·>100行·用户说严谨 |

**只链不抄**：BE 链 API-xxx；FE 链 UI-xxx（+UID）；dev 禁映射表与 JSON 粘贴。权威结构 → [dev-rationale](../templates/dev-rationale.md)。

### 分档判定时机

- **阶段 3 拆 todo 时**：为每个 T-xxx 标注档位（`### T-xxx：[端] 描述（估时）— F-001 [精简|标准|完整]`）
- **阶段 4 写 dev 前**：复检档位是否仍准确（编码中可能发现风险升高 → 升档）
- **默认档 = 标准档**；不确定时取标准档；用户可在 todo.md 覆盖

### 与快速/严谨模式的关系

| | 精简 | 标准 | 完整 |
|--|------|------|------|
| **快速/严谨** | 全档 摘要+步骤+结果；完整仅多字面量严检。快速不改段数 |

> 「全部做」= 催进度，不改档、不跳闸门。

### 升档规则

编码中发现风险升高（如：原来判定精简档，但实际涉及 DB）→ **当前②完成后升档**：
1. 更新 todo.md 中该 T 头的档位标注（如 `[精简]` → `[完整]`）
2. 补齐缺失段（精简→标准：补摘要五 bullet 并补步骤；标准→完整：须≥3 步 + 字面量）
3. 若升到完整档：补跑 `--gate dev-step1-literal --dev-file {该T的dev文件}`
4. 重新过可运行闸门（新档位可能有额外要求）
5. **已勾①**：不需重新勾，但须确认补齐的内容符合构思闸门标准

**禁止**降档（已写的段不删）。**禁止**不改 dev 就继续；**禁止**升档后跳过新增闸门。

## 闸门（按风险档分流）

```
Read exemplar → 写 dev →【写码前检查】→ 勾① → ② →【可运行闸门】→ ③
                  ↑ 精简/标准档合并为 1 道      ↑ 完整档拆为构思闸门+写码闸门 2 道
```

| 档 | 写码前 | 写码后 |
|----|--------|--------|
| **精简档** | 1 道「写码前检查」（合并构思+写码闸门） | 可运行闸门 |
| **标准档** | 1 道「写码前检查」（合并构思+写码闸门） | 可运行闸门 |
| **完整档** | 2 道：构思闸门 → 勾① → 写码闸门 | 可运行闸门 + 字面量 grep |

> **写码前检查**（精简/标准档合并闸门）分两步执行：
> 1. **构思部分（勾①前）**：文件存在 + 做法可执行 + 段标题齐全 + 非事后补写 + 读范例 → 通过后**勾①**
> 2. **写码部分（勾①后）**：①已勾 + 单任务约束 + 读过写法锚点 + 复用就绪 → 通过后**进入②**

### 构思闸门 — 勾 ① 前

> 权威检查表（9 项含「不过→」列）与字面量速查 → **[dev-quickstart §构思闸门](../templates/dev-quickstart.md#构思闸门勾-①-前)**。本文件不再复述表格，避免双维护漂移；改检查表只改 dev-quickstart 一处。

### 写码闸门 — Write 源码前

> 权威检查表 → **[dev-quickstart §写码闸门](../templates/dev-quickstart.md#写码闸门write-前)**。本文件不再复述，避免双维护漂移。

### 编码中调整（② 写码时发现 ① 需要改）

> **允许迭代，不死守原设计。** 写码是设计验证的过程——发现原构思有偏差是正常的。  
> 全阶段纠偏阶梯 → [change-management §纠偏阶梯](change-management.md#纠偏阶梯全阶段)（本表 = **L0**；model/sol 小改 = **L1**；动 AC = **L2**）。

| 情况 | 动作 | 需要回 ①？ | 级 |
|------|------|-----------|----|
| `## 步骤` 某个 `####` 细节调整 | dev 追加 `## 设计调整` ≤5 行，继续 ② | ❌ 不回 | L0 |
| `## 步骤` **根本性变更** | 回 ① 重写步骤 → 过写码前检查 → 再 ② | ✅ 回 ① | L0↑ |
| 发现 model/sol 需要小改且 AC 不变 | 见下方「轻量反馈回路」 | ❌ 不回 ① | L1 |
| 发现已确认 AC/实体关系错 | → [change-management L2](change-management.md#纠偏阶梯全阶段)；**禁止**用设计调整掩盖 | — | L2 |

**`## 设计调整` 格式**：
```markdown
## 设计调整（② 编码中发现）
- 原步骤 2 用 XxxService.create()，实际发现需先调 YyyService.check() → 已补步骤 2b
- 原步骤 3 合并为一步（两步逻辑不可分）
```

**禁止**：用「设计调整」塞入根本性方案变更以逃避回 ①（如整个做法重写却标「调整」）。发现即违规，须回 ①。

### 可运行闸门 — 勾 ③ / 父任务 ✅ 前

| 检查项 | 通过标准 | 不过 → |
|--------|----------|--------|
| **编译构建** | 本 T 涉及端 **编译/构建**成功（静态检查+构建） | 修编译，禁止勾③ |
| **能启能调** | 按 [能启能调分档](../templates/dev-quickstart.md#能启能调分档)：**该端第 1 个 T 须实际启动**；后续单 T 可 MockMvc；**F/MVP 切片与给用户看须真实启动** | 修启动与配置 |
| **主路径冒烟** | 本 T 主路径不炸。主路径 = `## 步骤` 第一个 `####` 涉及的接口/页面 | 修功能 |
| **证据已写** | 证据已写入 dev `## 结果` 或 todo（命令+结果） | 补证据 |

权威细则与「给用户看 / 模块级」时机 → [dev-quickstart 可运行闸门](../templates/dev-quickstart.md#可运行闸门每-t--每模块强制)。

### 与阶段 5 入场门禁的关系（合并验证）

> **本闸门 = 唯一可运行验证节点。** 阶段 5 入场门禁不再全量重跑，改为增量模式。

| 场景 | 阶段 5 入场门禁动作 |
|------|---------------------|
| **同会话**内可运行闸门已过（todo checkpoint 有时间戳） | 只验**增量**：本次变更涉及的模块集成冒烟；不重跑已验证端 |
| **跨会话**续作（距上次可运行闸门 >1 天或新会话） | 全量重验：编译→启动→冒烟（现行规则） |
| 可运行闸门证据缺失或不可复现 | 全量重验 |

阶段 5 仍负责 **AC 验收归档 + 全量回归**，但入场门禁与阶段 4③ 合并。

## ①②③ 要点

- **①** 模板 → [dev-rationale](../templates/dev-rationale.md)；照 exemplar 颗粒度
- **②** 按 `## 步骤` 写码；FE 字段对齐 **UI-xxx §字段绑定**
- **③** 先过 **可运行闸门**，再对照 REQ AC 写 test/ac；`## 结果` 回填（含可运行证据）；父任务 ✅

### 轻量反馈回路（= 纠偏 L1）

> dev → model/sol 待回溯。**dev 里发现 model/sol 需要小改时，不要每次都走 5 步变更管理。** 先记笔，后批量回溯。  
> 权威 → [change-management §纠偏阶梯](change-management.md#纠偏阶梯全阶段) · [§轻量反馈回路](change-management.md#轻量反馈回路)（含对照表 + 冲突裁定：同时命中轻量+完整→按完整处理）

### 默认串行（禁止批量① / 禁止 Subagent 外包）

> **阶段 4 入口代码基线检查**：若 sol 确认后用户手动改了代码（git hash 变化），AskQuestion「检测到方案确认后代码已有变更。是否更新方案？(是→回阶段3 / 否→基于当前代码继续)」→ 停。
> **大型项目分批（T ≥ 20）**：当前批次 ±5 个 T 展开①②③；已完成批次折叠为 `已完成: T-001~T-015 ✅`；未开始折叠为 `待做: T-021~T-050`；按 F-xxx 切片分批。

0. **TodoWrite 展开**（TodoWrite强制展开）：为每个 T 建 **①②③ 三条**（强制）；「全部开发」时①条目须先全部出现为 pending  
1. 取**一个** ①未勾 的 T → 进行中  
2. 读范例→写该 T 的 dev→构思闸门→勾①（TodoWrite + atlas/todo）→写码闸门→②→**可运行闸门**→③  
3. **在该 T 的 ② 开始前，禁止**再为其他 T 写 dev 或勾①（**显式并行批量构思 除外**，见下）  
4. 「全部开发」时重复 1–3，**仍由主 Agent 串行**；禁止用 1～2 个 Subagent「实现全部 BE/FE」；**禁止跳过可运行闸门**  
5. 全部父任务 ✅ **且** `dev` 文件数 = T 头数 **且** 每个 T 的 TodoWrite① completed **且** 可运行闸门 证据齐 → 结束闸门 → 停  
6. 任一 **F-xxx / MVP 切片** 相关 T 全 ✅ → 模块编译+冒烟（可运行闸门）→ **`user_decide`**：阶段性确认卡 → 停；**`AF_DECIDE=ai`**：默认继续下一 F（用户说「先给我看/演示」才发卡）→ 见 [askquestion-gate §阶段性确认](../templates/askquestion-gate.md#阶段性确认卡阶段-4-内--mvp--f-xxx-切片)
7. 用户要「给用户看」或选「先给我看」→ 存在端全量启动冒烟通过 + 给出启动步骤；等用户反馈  
8. 全部 T ✅ → 更新 `atlas/README.md`（现在/阻塞）→ 结束闸门（AI自主→审阅闸门；user_decide→阶段闸门；进 tests）→ 停；prompt 须含可运行闸门 证据摘要

### 显式并行时（与「串行约束」 的关系）

仅用户原话含「并行 / subagent / 同时开 FE+BE」时启用 → [parallel-orchestration.md](parallel-orchestration.md)：

| 批次 | 允许 | 仍禁止 |
|------|------|--------|
| B（批量勾①） | 多个 T 先各自写合规 dev、过 A、勾① | 「全部开发」冒充并行；无①就② |
| C（并行②） | 1 Task = 1 T，且该 T ①已勾 | 一个 Subagent 多 T；Subagent 写①或改 todo ✅ |

**① 只由主 Agent 写**；Subagent **只做②→可运行闸门→③**。
