# dev 阶段规则说明书

> 对应实现：`index.mjs`（聚合入口） + `steps.mjs`（步骤解析工具）。  
> 权威流程规范另见 `phases/04-development.md` 与 `templates/dev.md`。

## 1. dev 文档到底要回答什么问题

> **FE/MP 主形态（v9.20+）**：`## 主流程` + `## 边界` + `## 实现说明` — 见 [dev-granularity](../../../templates/dev-granularity.md)  
> **BE 主形态**：`## 步骤` **4 列流程表**

每个 `atlas/dev/T-xxx-*.md` 不是「写码日志」，而是**写码前的构思契约 + 写码后的验收证据**。它必须让另一个人（或 30 天后的自己）能不看聊天记录就回答：

1. **本 T 是谁？** → `## 摘要` 里的 **本 T**；
2. **边界在哪？** → `## 摘要` 里的 **做 / 不做 / 上游** + `## 边界`（FE/MP）；
3. **怎么验收？** → `## 摘要` 里的 **AC**；
4. **具体改哪？** → FE/MP：`## 实现说明`；【新写/改动】+ 落点；BE：`## 步骤`；
5. **跑通没？** → `## 结果` 里的编译 + 启动/冒烟 + exit0/✅/PASS。

## 2. 每个检查规则是干什么的

### 2.1 结构类（先保证文档有骨架）

| 规则 | 报错时机 | 为什么要这样 |
|------|----------|--------------|
| `DEV-SEC-摘要` | 缺少 `## 摘要` | 没有摘要，读者无法 3 秒定位这个 T 在做什么。 |
| `DEV-SEC-步骤` | BE 缺少 `## 步骤` | BE 没有步骤，等于没有写码前的具体计划。 |
| `DEV-SEC-flow/edge/impl` | FE/MP 缺少主流程/边界/实现说明 | 叙述五段式不完整。 |
| `DEV-SEC-结果` | 缺少 `## 结果` | 没有结果，等于没有验收证据。 |
| `DEV-FAKE-标题` | 用纯文本「范围：」而不是 `## 摘要` | 假标题无法被脚本/人类稳定识别。 |
| `DEV-FAKE-过短` | 文档去空格后小于 500 字 | 防止用一句「实现登录」糊弄过去。 |

### 2.2 摘要类（边界 + 上游 + 验收）

| 规则 | 报错时机 | 为什么要这样 |
|------|----------|--------------|
| `DEV-SUMMARY` | 没有 `## 摘要` | 同上。 |
| `DEV-SUMMARY-定位` | 摘要里没写 **本 T** | 必须明确这个 T 对应哪个 F-xxx / REQ-xxx，避免「实现登录」这种含糊描述。 |
| `DEV-SUMMARY-结构` | 缺少 **本T/做/不做/上游/AC** 五类 bullet | 五类 bullet 是防范围蔓延和验收扯皮的最小信息集。 |

### 2.3 步骤类（把「做」拆到可执行粒度）

| 规则 | 报错时机 | 为什么要这样 |
|------|----------|--------------|
| `DEV-STEP-最少` | 步骤数少于 3 | 防止「一步写完」导致思考粒度太粗。 |
| `DEV-STEP-FULL-须步骤表` | 没用 4 列流程表 | BE 须用 S1… 流程表，注意点含落点与分支。 |
| `DEV-BAN-ATOM` | 用了 8 字段原子步骤表 | 已废弃；须改用 4 列流程表。 |
| `DEV-BAN-HASH` | 用了 #### 薄写 | 已废弃；须改用 4 列流程表。 |
| `DEV-BAN-FLOW-5COL` | 用了 5 列含「目的」流程表 | 只用 4 列。 |
| `DEV-BAN-FE-步骤` | FE/MP 出现 `## 步骤` | FE/MP 须用 主流程+边界+实现说明。 |
| `DEV-STEP-流程落点` | 流程表「注意点」栏没有代码落点 | 步骤不能飘在天上，必须落在具体类/方法/路径上。 |
| `DEV-STEP-FLOW-COLS` | 数据行与表头声明列数不一致 | 防止列错位。 |
| `DEV-LIT-代码落点` | 全文没有 `Class.method` / `path/` / `func()` 反引号 | 防止整个文档都是业务描述，找不到具体代码位置。 |

### 2.4 链路与 SSOT 类（只链不抄）

| 规则 | 报错时机 | 为什么要这样 |
|------|----------|--------------|
| `DEV-LINK-API` | BE dev 没有链 API-xxx | 后端实现必须对应已确认的契约，不能私自改接口语义。 |
| `DEV-LINK-UI` | FE dev 没有链 UI-xxx / UID-xxx / F-xxx | 前端实现必须对应已确认的设计与边界。 |
| `DEV-LINK-UI-API` | FE dev 链了 API 但没链 UI-xxx | 调 API 时，字段绑定以 UI-xxx 为准，不能绕过 UI 契约直接看 API。 |
| `DEV-BAN-映射` | dev 内出现字段映射表（表头含「字段」+「映射/后端字段/接口字段」） | 字段映射是 contracts/UI 的 §字段绑定 的职责，dev 只链。 |
| `DEV-COPY-JSON` | dev 内粘贴大段 JSON 契约（含多个 `"key":` 键值对） | 契约 JSON 应该存在 contracts/API，dev 只链。 |
| `DEV-BAN-联调卡` | dev 内出现「联调卡」 | 联调是开发后的协作，不应该塞进单个 T 的 dev 文档。 |

### 2.5 旧形态与冗余类（保持 v9.11 统一格式）

| 规则 | 报错时机 | 为什么要这样 |
|------|----------|--------------|
| `DEV-BAN-范围` | 出现 `## 范围` | 范围信息应并入 `## 摘要` 的 **做/不做**。 |
| `DEV-BAN-异常` | 出现 `## 异常` | 异常处理应作为步骤里的 **系统** 行或引用 AC-xxx。 |
| `DEV-BAN-AC` | 出现 `## AC` | AC 应放在摘要 bullet 和结果映射表，不要单独成节。 |
| `DEV-BAN-旧标题` | 出现 `## 一、目标` 或 `## 五、可执行方案` | 统一用 `## 摘要 / ## 步骤 / ## 结果`。 |
| `DEV-BAN-构思章` | 出现 `## ① 构思` | 构思状态由 todo.md 的 checkbox 体现，不要重复成节。 |
| `DEV-BAN-关键实现点章` | 出现 `## ② 关键实现点` | 统一用 `## 步骤`。 |
| `DEV-LEGACY-段` | 出现 `## 前置 / ## 必读 / ## 契约` | 这些旧段名已收敛到 `摘要/步骤/结果` + 链 sol。 |

### 2.6 结果与验收类（证明它真的跑通了）

| 规则 | 报错时机 | 为什么要这样 |
|------|----------|--------------|
| `DEV-AC-MAP` | `## 结果` 里缺少 AC 映射表（表头须含 `AC` / `AC ID` 与 `unit/验证/方法/方式/test/人工/冒烟` 任一列） | 勾 ③ 前必须明确每个 AC 是通过单测、人工还是冒烟验证的。 |
| `DEV-AC-UNIT` | BE 的 AC 映射表没有 `test/unit` 路径 | 后端建议一个 AC 对应一个单元测试，形成可重复验证。 |

## 3. 步骤写法（按端）

| 端 | 形态 | 何时 |
|----|------|------|
| **FE/MP** | 叙述五段式：主流程 + 边界 + 实现说明 | 默认；含 `-FE.md` / `-MP.md` / `[FE]` / `[MP]` |
| **BE** | **4 列流程表** `S1…` | 默认；禁 8 字段原子表 / #### 薄写 / 5 列含目的 |

### FE/MP 叙述五段式

详见 [dev-granularity.md](../../../templates/dev-granularity.md) · 范例 [dev-exemplar-FE.md](../../../examples/dev-exemplar-FE.md)。

| 段 | 校验规则 |
|----|----------|
| `## 主流程` | `DEV-FLOW-入口` · `DEV-FLOW-最少` · `DEV-FLOW-落点` |
| `## 边界` | `DEV-EDGE-存在` |
| `## 实现说明` | `DEV-IMPL-块` · `DEV-IMPL-字段` · `DEV-IMPL-落点` |

### BE 4 列流程表

范例 [dev-exemplar-BE.md](../../../examples/dev-exemplar-BE.md)。

```
| 步骤 | 动作 | 输入 → 输出 | 注意点（含落点） |
| **S1** | … | … → … | 新写 `Class.method` — 分支/错误码 |
```

| 列 | 回答什么 | 常见错误 |
|----|----------|----------|
| **步骤** | 第几步 | 用文字「第一步」代替 `S1` |
| **动作** | 这步做什么 | 写成需求描述 |
| **输入 → 输出** | 进出数据 | 只写输入不写输出 |
| **注意点（含落点）** | 代码落点 + 分支/坑 | 只写业务说明不写 `` `Class.method` `` |

注意点列里的五种说法速查：

| 说法 | 什么时候用 |
|------|-----------|
| **继续走** | 本步不改已有代码，靠现有逻辑跑 |
| **在 X 上加** | 扩展现有类/文件 |
| **直接用** | 零改动引用已有组件/工具 |
| **照 Y 那套** | 对齐邻居文件/同模块已有写法 |
| **新写** | 须新建文件/类 |
