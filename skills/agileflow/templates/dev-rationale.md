# 构思落盘文档（atlas/dev/）

> 阶段 4 **步骤 ①**。  
> **dev = 构思落盘，不是事后补文档。**  
> **没落盘 → 禁止写业务源码。**

---

## dev 规则：构思先行

**开发的第一件事是构思，第二件事才是写码。** ① 须按下方**七段模板**完整输出到 `atlas/dev/`。

| 规则 | 说明 |
|------|------|
| **D1** | dev 文档 = **构思落盘**；聊天里的思路不算，必须写文件 |
| **D2** | 一任务一构思：`T-{id}-{简述}-{BE\|FE}.md` |
| **D3** | **七段齐全**（一～七）；缺一段 = 没想全 |
| **D4** | **五、核心流程**是构思核心：逐步编号，接手人能照着写码 |
| **D5** | ② 只执行 **五**；③ 只验证 **八**；**顺序 ①→②→③ 不可跳** |
| **D6** | 写码后再补 dev = **无效构思** |
| **D7** | `atlas/todo.md` 每 T-xxx **须有三步子项**；**① 未勾 → 禁止写码** |

architecture / contracts 讲**是什么**；**dev 讲这任务怎么构思、怎么写、怎么验**——不可互相替代。

细则：[04-development.md](../phases/04-development.md)

## 七段 → ②③ 怎么用

| 段 | 章节 | 构思时必答 | ②③ 怎么用 |
|----|------|------------|-----------|
| 理解 | **一** | 核心目标、必须满足、**明确不做** | ② 不越界 |
| 数据 | **二** | 表/实体/字段/约束 | ② 建表、Entity |
| 契约 | **三** | 接口、入参、出参、异常码 | ② 实现 API |
| 状态 | **四** | 状态流转（无则写「无」） | ② 状态校验 |
| 流程 | **五** | **逐步主逻辑** | ② **严格按序写码** |
| 边界 | **六** | 并发、幂等、异常场景 | ② 异常处理 |
| 依赖 | **七** | 框架、中间件、假设、humanTodo | ② 按栈实现 |
| 验收 | **八** | REQ AC + test/ac 规划名 | ③ 只跑此表 |
| 结果 | **九** | 表头 ① 必建 | ③ 回填 L3 |

**快速模式**：**一、五、八不可省**；二/三/四/六/七 按任务类型裁剪（纯 FE 二改组件状态）。

- 任务：**T-002**（唯一）
- 端：**BE**

## 一、需求理解（我理解的你要什么）

- 核心目标：用户下单后启动 30 分钟倒计时，超时未支付则自动关单并回滚库存
- 必须满足：精准计时、库存回滚幂等、并发安全、不误关已付款订单
- 明确不做：支付回调的具体实现（只留接口）、退款逻辑、消息通知

---

## 二、数据模型（需要哪些表/实体）

| 实体 | 关键字段 | 类型 | 约束/说明 |
|------|---------|------|----------|
| order | id | BIGINT | PK |
| | order_no | VARCHAR(32) | 唯一索引 |
| | status | TINYINT | 1:待付款 2:已付款 3:已取消 4:已超时关闭 |
| | create_time | DATETIME | |
| | expire_time | DATETIME | 创建时间+30分钟 |
| | sku_id | BIGINT | |
| | quantity | INT | |
| sku | id | BIGINT | PK |
| | stock | INT | 必须 ≥ 0 |
| | version | INT | 乐观锁版本号 |

---

## 三、接口契约（对外暴露什么）

| 接口 | 方法 | 入参 | 出参 | 异常码 |
|------|------|------|------|--------|
| 创建订单 | POST /orders | skuId, quantity, userId | {orderNo, expireTime, payUrl} | 400:库存不足 409:重复提交 |
| 查询订单 | GET /orders/{orderNo} | orderNo | {orderNo, status, expireTime} | 404:订单不存在 |
| 支付回调 | POST /orders/{orderNo}/pay | orderNo, payResult | {success} | 409:订单状态不符 |

---

## 四、状态机（如果有状态流转）

```
[待付款] --支付成功--> [已付款]
[待付款] --超时(30min)--> [已超时关闭]
[待付款] --用户主动取消--> [已取消]
```

**约束：只有 [待付款] 能触发超时关闭，已付款/已取消的订单禁止任何变更。**

---

## 五、核心流程（主逻辑时序/步骤）

**创建订单：**
1. 校验 skuId 存在、quantity ≥ 1
2. 乐观锁扣减库存：`UPDATE sku SET stock=stock-?, version=version+1 WHERE id=? AND stock>=?`
3. 若更新失败 → 抛「库存不足」
4. 生成订单，status=待付款，expire_time=now+30min
5. 将 orderNo 推入延迟队列（Redis ZSET / RabbitMQ DLX），score=expire_time

**超时关单（定时任务扫描 / 延迟队列消费）：**
1. 取出到期的 orderNo
2. 查订单状态，若已不是「待付款」→ 直接丢弃（幂等）
3. 开启事务：
   - 更新订单 status=已超时关闭
   - 回滚库存：`UPDATE sku SET stock=stock+? WHERE id=?`
4. 记录操作日志

**支付回调：**
1. 查订单，若 status ≠ 待付款 → 返回「状态不符」，不处理
2. 更新 status=已付款
3. 从延迟队列中移除该 orderNo（或消费时二次校验）

## 六、异常与边界（我会处理这些坑）

| 场景 | 处理方式 |
|------|----------|
| 用户重复点击创建订单 | 前端防抖 + 后端 userId+skuId 分布式锁（5秒） |
| 定时任务宕机期间漏扫 | 启动时补偿扫描：expire_time < now AND status=待付款 |
| 库存回滚时 sku 被删除 | 记录异常日志，人工介入，不抛异常阻断流程 |
| 支付回调和超时关单并发 | 数据库行锁（SELECT FOR UPDATE）或状态 CAS 更新 |
| 订单已取消，支付回调才来 | 幂等校验：status 不是待付款则拒绝，触发退款（V2） |

## 七、技术选型与依赖

- Spring Boot 3.x + MyBatis
- MySQL 8.0（事务隔离级别：READ COMMITTED）
- Redis（分布式锁 + 延迟队列 ZSET）
- 定时任务：Spring @Scheduled 每 30 秒扫描兜底
- 关键假设：支付回调接口由外部系统提供，本系统只接收结果
- humanTodo：无
