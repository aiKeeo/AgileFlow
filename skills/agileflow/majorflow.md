# AgileFlow 核心思想

> 把想法拆成可执行阶段，落到可运行交付——**流程本身就是产品**。
> **定位**：给人看的产品说明（**本文件即 skill 内 SSOT**）。仓库根 [majorflow.md](../../majorflow.md) 为短链入口。Agent 运行时入口是 `SKILL.md`；编排实例是项目 `atlas/flow.yaml`。

---

## 产品是什么

AgileFlow 要解决的不是「让 AI 多写点代码」，而是：

**把想法拆解到实现落地的一套流程。**

流程即产品。每个阶段有固定规范：先把业务与设计说清楚，再构思，再写码，再验收。

防 AI 幻觉、不听指挥、假装完成，是后来加上的加固——**服从拆解，不反客为主**。

用户把决策权交给 AI 时，摩擦可以少，证据不能少。「你定 / 别问我」= 少澄清、少审阅卡；≠ 偷偷关阶段、≠ 空跑勾选、≠ 先码后补。

**质量跟 adoption**：`atlas/flow.yaml` 里**未 skip 且走过**的步骤，才验、才准宣称；已 `skip` 的不验、不准吹。

## 两轨

| 轨 | 怎么进 | 管什么 |
|----|--------|--------|
| **快捷轨** | `/af-fix` / `/af-refactor` / `/af-tweak` / `/af-perf` / `/af-chore` / `/af-ut` / `/af-revise` | 小改；零文档或修订**已有**；无确认卡、无主链闸门、不派角色。越界 → 升级完整轨。细则 → `skills/agileflow/phases/quick-commands.md` |
| **完整轨** | 正式交付意图（无上述前缀，或升级后） | 按 `atlas/flow.yaml` 走**未 skip** 的步；默认骨架即下方主链。**新增/重做阶段**（含插入步）= 改 flow 并从该步**重新走**，不是 revise |

未启用 AF 时的微型/hotfix/问答 **豁免** 仍存在；**已启用 AF 后**轻量改动走快捷轨，不靠装豁免。

## 管辖边界（铁律）

**只有 `atlas/flow.yaml` 的 `steps[]` 里的 id 才受 AgileFlow 流程管理**（`AF_STEP`、skip、depends/outputs、主链闸门、派活台账、advanceStep、质量宣称）。

| scope | 门牌 | 受 flow 管理 | 说明 |
|-------|------|-------------|------|
| **flow** | `/af-req` … `/af-test` + 自定义插入步 | **是** | 默认主链五步；`/af-tests` = `/af-test` 别名 |
| **pre-flow** | `/af-init` | **否** | brownfield 盘点；`init-confirm` 后进第一个 flow 步；**永不**写入 flow steps |
| **routing** | `/af`（默认）、`/af-explore` | **否** | `/af` = 万能自动路由；`af-explore` = 窄探索支路；不写 env/REQ/flow（落地到真实步后另说） |
| **quick** | `/af-fix` … `/af-revise` | **否** | 快捷轨；越界 → 升级 flow 轨 |
| **豁免** | （无门牌） | **否** | 仅未启用 AF 的问答/微型/hotfix |

脚本 [`FLOW_RESERVED_IDS`](scripts/validate-atlas/lib/flow.mjs) 禁止 init/explore/**af**/快捷 id 进入 flow.yaml。

## 编排：`atlas/flow.yaml`（与 todo 切割）

| 文件 | 职责 |
|------|------|
| **`atlas/flow.yaml`** | **`steps` = 默认执行顺序链条**；每步：`mode` / `prompt` / **`depends`** / **`outputs`**；orch 加 `criteria`；按需 `skip` |
| **`atlas/todo.md`** | T 与 ①②③ |
| **`atlas/agileflow.env`** | **`AF_STEP`**（进度光标）+ **`AF_PHASE`**（闸门档，由 AF_STEP 派生） |

- **默认：按 steps 序执行；`AF_STEP` 跟踪当前步或并行波（`id1,id2`）。** 无互依前置的步可同波并行；门牌 = step `id`（打 `/af-research` 即进该波）。完成或合法 skip 后进下一波。
- **可跳的唯二情况**：① 该步是 **`orch`**，AI 对照 **`criteria` 判定可跳**；② **用户明确要求**跳过（含关严格步）。都须落盘 `skip` + `reason`。
- **`strict`**：无用户明示 → AI **无权** skip。禁止静默跳、禁止赶工跳。
- **路径写在模板每一步的 `depends`/`outputs`**，不另臆造字段。无 `by`。
- Schema+默认全文 → `skills/agileflow/templates/flow.md`。

### Run 与当前证明

完整轨每次变更对应一个 Run，状态位于扁平目录 `atlas/runs/<runId>/`。阶段收口固定顺序：

1. `agileflow artifact scan --root .`
2. `agileflow log --door /af-… --summary … --route … --root .`
3. `agileflow gate --gate <gate> --root .`
4. `agileflow run gate-status --gate <gate> --root .`
5. 仍为 `PASS (pass)` 才 `agileflow step sync --root .`

有 current Run 时 JSONL Runtime 回执是唯一权威；legacy Markdown PASS 不参与判断。该步产物、attempt 或 flow 改变会使对应 PASS 失效；任一已登记产物 dirty 会阻断新 gate。flow 改变必须 abandon 旧 Run 后新开，禁止原 Run 偷刷 digest。

## 主链（完整轨默认骨架）

```
af-req → af-mod(按需) → af-sol → af-dev(①②③) → af-test
```

brownfield 另有前置 `/af-init`（盘点已有代码与业务）：**进场初始化，不进 `flow.yaml` steps**；做完再进主链。

| 阶段 | 一句话 |
|------|--------|
| **af-req** | 把「要什么」写成可独立验收的需求 |
| **af-mod** | 把「业务世界里有什么」说清楚——没有则几乎无视 |
| **af-sol** | 定边界、暴露面、技术摆法，拆成带 ①②③ 的开发任务 |
| **af-dev** | 先盘可复用资源再构思；写法给人看得懂；写码；按 AC 证明做对了 |
| **af-test** | 证明整批可交，而不是单测绿就算完 |

**续跑**：`user` 模式在阶段闸门后等人说「继续」；断点可用「继续 agileflow」。
**`AF_DECIDE=ai`**：闸门绿后须同会话连做到终点（阻塞式派 Subagent），禁止派完一批等人说「继续」。

## 各阶段

### req — 需求澄清

**解决什么**：交付什么业务能力；怎样才算验收通过。

- 需求是**验收的唯一权威**。AC 用 Given/When/Then 写在 REQ 里；下游只引用，不另造验收故事。
- **一功能一份 REQ**。MVP 是范围标签，不是把多功能力揉进一份总览。
- 有界面时结构与交互在 UID；链了 UID 就必须有文件，禁止空链标「已确认」。
- 信息够清楚就少问；没说清的只问缺口。问清楚是为了落盘，不是为了多聊。

**结束时应有**：已确认的 REQ（+ 按需 UID），人知道「做什么 / 怎么验收」。
**不要变成**：技术方案、接口明细、开发任务清单，或 REQ 挂 UID 目录却空。

---

### model — 数据建模（按需）

**解决什么**：业务实体、关系、规则、必要时的持久化形状。

- **看项目，不看仪式**。复杂实体/状态机 → 加重；简单 CRUD、无新实体 → **可几乎无视**。
- 「几乎无视」≠ 静默跳过：该步须为 **`mode: orch`** 且对照 **`criteria`**；总控写 `skip` + `reason` 后再决定派不派 `prompt: role-model`。
- 只回答业务世界是什么，**不**定义 API、不拆开发任务。

**结束时应有**：可用 model，或 flow 上白纸黑字的总控跳过判定。
**不要变成**：每个小改动都强制全量建模；角色自己改 flow 跳过；或跳过成本与全量一样重。

---

### sol — 方案设计

**解决什么**：系统做哪些功能、边界在哪、对外暴露什么、技术怎么摆、开发按什么顺序做。

- 方案是**从需求到任务的桥**：F 说边界与暴露面；contracts 承载接口/页面规格；架构全局一份；任务进 `todo`（每任务 ①②③）。
- **F 极简**：边界 + 暴露面；联调细节、字段映射、验收复述各有权威源，下游只链不抄。
- 技术栈先有来源（人定 / AI 记 / 仓库已有），再写架构。
- 任务默认串行；`AF_DECIDE=ai` 的正当加速 = **并发作业**，不是薄写 F/契约/todo。
- todo 里 ①②③ 此时仍应未勾选——清单是承诺形状，不是完成证明。

**结束时应有**：可执行的功能边界 + 契约（按需）+ 架构 + 带档位的开发任务清单。
**不要变成**：抄 REQ、提前写源码与测试报告、提前勾满 ①②③，或因「少停」写成空壳。

---

### dev — 开发实现

**解决什么**：把方案里的每个任务真正做出来，跑得起来、对得上 AC，且人看得懂 AI 怎么做。

- 每任务固定三步：**① 构思 → ② 写码 → ③ 验 AC**。无合规构思，禁止写业务源码。
- **勾选 = 承诺已兑现**：勾① 须有 `atlas/dev/T-*.md`；勾③ 须「## 结果」里有可运行证据。
- **① 叙述五段式**（唯一质量线 = 完整）：摘要 + 主流程 + 边界 + 实现说明 + 结果。逻辑块「怎么做」须编号 ≥2；边界 ≥2 且挂「第 N 步」。接现有代码写清继续走 / 在…上加 / 照… / 新写。细则 → `skills/agileflow/templates/dev-granularity.md`。
- **可运行先于「写完」**：编译 / 能启 / 本任务冒烟——证据进构思「结果」；单测绿 ≠ 可交。
- 默认一次推一个任务；要快 → **并发**（`ai` 可推并行启动卡；`user` 须人显式要）。

**结束时应有**：每个 T 有可读构思、有代码、有可运行证据、AC 对得上。
**不要变成**：先码后补构思、一行糊弄、无视库存造轮子、用 checkbox 全绿代替真的跑过。

---

### test — 测试验收

**解决什么**：整批功能能否交付；质量与回归是否站得住。

- 阶段 4 的 ③ 已按 AC 写测并跑过；阶段 5 **不重写测试**，负责入场、归档、回归。
- **入场是 ③ 的延续**：同会话复用已有证据，只补跨任务集成；跨会话或证据丢了才全量重验。
- 单测绿、分层点名跑通，都不能单独冒充「全量可交」。

**结束时应有**：验收报告齐、回归结论清楚（PASS / 阻断 / 失败）。
**不要变成**：同会话无意义三重复跑，或缺测却硬标完成。

---

### init — 项目盘点（brownfield，主链外）

从零新建**不做** init；有存量才盘点。盘点是为了后续少幻觉、少撞命名，不是第四条主链阶段。

## 委托交付（`AF_DECIDE=ai`）

常见原话：「你定」「别问我」「全程你看着办」。未说清谁决策时仍须启动卡问人。

| 可以少 | 不能少 |
|--------|--------|
| 阶段内澄清卡、技术栈追问 | `flow.yaml` 中**启用**步的顺序与落盘；未启用步不装完成 |
| 阶段结束少停（A 档绿 → 一行摘要） | todo ①②③、构思文件、可运行证据、REQ AC 回填（当对应步启用时） |
| 等人说「继续」才开下一批 | 启用步闸门 exit 0；阻塞派活同会话循环到交付 |

**易混**：「你定」≠ 已答启动卡；≠ 擅自关 req/test；≠ 豁免捷径 / 先码后补。关核心步须用户明示；model 按需跳过须总控写进 flow。AI 自主少问时，形柱与跑柱必须更硬，否则必偷懒。

---

## 加固：形 / 令 / 跑

AI 容易幻觉、不听指挥、空跑冒充完成，于是加了三层加固——**概念对等，执行有硬有软**：

| 柱 | 管什么 | 靠什么 | 约束力 |
|----|--------|--------|--------|
| **形** | 按阶段写对；勾选对得上文件；假证据过不去 | `atlas/` + `validate-atlas`（`TODO-CHECK-*`、`REQ-UID-断链` 等） | **硬阻断** |
| **令** | 听不听指挥；阶段能不能往下走 | 首启启动卡；`user` 阶段停点；`ai` 少停但须自治连做 | 纪律 |
| **跑** | 真能编译 / 启动 / 冒烟 | 可运行闸门；证据写入 `## 结果`；脚本不替你跑 | **硬阻断** |

委托交付下令柱变软时，靠形柱/跑柱硬挡补上——否则「全让 AI 干」就会空跑勾选、假装完成。

## 配套机制

| 机制 | 作用 |
|------|------|
| **流程编排（flow.yaml）** | 完整轨启用哪些步；总控改、角色不改；质量跟启用 |
| **决策维（ai/user）** | 只控停点与是否推并发；不减 sol/dev 质量；不替用户关核心步；不取消①或可运行闸门 |
| **质量线** | dev 唯一完整档：叙述五段式 + 逻辑块怎么做编号；`AF_TIER=full` 不减厚度 |
| **快捷轨** | 已启用 AF 亦可用的轻量通道（非豁免） |
| **豁免** | 仅未启用 AF 的问答/微型/hotfix；灰区 AskQuestion |
| **CLI / 门牌 skill** | `npx @agileflow/cli init` 装 `skills/agileflow` + `skills/af-*`；改 flow 后 `update --step-skills-only`；闸门 `agileflow gate` |
| **薄总控 / 厚角色** | `SKILL.md` 只路由+裁决+≤15 红线；写法在 role layers / phases |
| **正确做法与红线** | 正向节奏优先；踩线列 ≤15 条（orchestrator SSOT），不堆禁令 |
| **纠偏 L0–L3** | 中途发现不对先判级再动作；禁止硬扛继续写码 |

**证据容器**：`atlas/`（requirements / model / solution / dev / tests 物理隔离）。业务源码写在工程正常位置。
**下游只链不抄**：验收看 REQ AC，线框看 UID，接口看 contracts，边界看 F。

## 非目标

- 不是「纯防幻觉工具」——防失控服从拆解
- 不做豁免场景的完整流水线（微型/hotfix/纯问答走豁免；小改走快捷轨）
- 不把编排塞进 `todo.md`，不用 `uses`/多档 preset 配置面糊弄用户
- 脚本不替用户编译运行——只验落盘之形与勾选是否对得上证据
- 默认串行；`ai` 可推并发，但「全部开发」≠ 自动启 Subagent
- 不把「AI 自主」做成薄文档或跳证据的快速通道
