# 流程模式：快速 vs 严谨

> **本文件权威范围**：快速/严谨下 **user_decide 的停点合并**、REQ 信息充分例外、连续做少卡、**快=并发不减文档**。  
> 跨切冲突以 [SKILL 裁决表](../SKILL.md#裁决表冲突时以此为准) 为准；dev 档位 → [04](../phases/04-development.md)；AI 停点 → [stage-delegation](stage-delegation.md) / [askquestion-gate](askquestion-gate.md)。  
> 首行声明须标注 `模式：快速` 或 `模式：严谨`。  
> **对照例子**（薄写 vs 同质+并发）→ [fast-means-parallel](../examples/fast-means-parallel.md)

## 铁律：快速 ≠ 跳阶段 ≠ 薄 sol/dev

> [SKILL 裁决表](../SKILL.md#裁决表冲突时以此为准) 权威。快速仍按序走完阶段、todo 仍详细；「用户不用管」= AI 自主 ≠ 快速。  
> **「快速模式」≠「快速通道」**：快速通道仅微型改动，见 [00-intent-routing](../phases/00-intent-routing.md#①-豁免判定最先做)。

### 快只体现在两件事

| 可以因「快速」变 | **禁止**因「快速」变 |
|------------------|----------------------|
| **停点更少**：`user_decide` 确认+闸门可合并；`fast+ai`+A 档绿免审阅卡 | sol / contracts / architecture / todo / **dev** 的颗粒度、段结构、可执行落点 |
| **并发作业**：阶段 4 有 ≥2 无路径冲突 T 时 **推并行启动卡**，用 Subagent 并行写码加速 | 用「待补齐 / 见代码 / 做登录相关」糊弄开发可读性 |
| 集成覆盖率不卡阈值（出报告即可） | 跳①、跳可运行闸门、合并多 T、空壳步骤 |

**两维正交（勿混）**：定义 → **[stage-delegation §两维模型](stage-delegation.md#两维模型)**（权威）  
**常见组合**：用户**不想自己决策** → **[委托交付 fast+ai](stage-delegation.md#委托交付fastai用户不想自己决策)**（`AF_FLOW=fast` + `AF_DECIDE=ai`）；用户另说「严谨」→ `strict+ai`。

> 速记：快速/严谨 控**停点 + 是否推并发**；AI自主 控选型要不要问用户。两维均**不**控制阶段顺序、todo 详细度、sol/dev 质量、能否跳①/可运行闸门。  
> **勿混**：「快速」 alone ≠ 不想决策；「你定」 alone（未说严谨）→ 默认委托交付 = fast+ai。  
> **要加速的唯一正当手段** = 并发作业（见 [parallel §快速模式推并发](../phases/parallel-orchestration.md#快速模式推并发)），不是减文档。

## 风险分档（与快速/严谨正交）

> **流程重量与任务风险成正比，不与任务数量成正比；也不与快速/严谨挂钩。**
> 三档定义（精简/标准/完整）、触发条件、校验器行为 → **[04-development §风险分档](../phases/04-development.md#风险分档)**（权威定义）
>
> 速记：小任务默认精简（单端·无DB·无权限/支付·<100行·无跨模块）；精简=摘要/步骤/结果 · 标准=摘要五 bullet · 完整=同标准+字面量严检；template 步骤须 **涉及改动**（legacy：`AF_TEMPLATE=no` 仍用 用户/系统/改）（格式权威 → [dev-quickstart §构思闸门](dev-quickstart.md#构思闸门勾-①-前)）  
> **快速模式不得把「标准/完整」任务压成精简档**；升档不降档。

## 决策委派（与快速/严谨正交）

**首启**：无 env 或 `AF_FLOW`/`AF_DECIDE=pending` → **必须**[流程启动卡](stage-delegation.md#流程启动卡首启强制) → 停。禁止静默默认快速+AI自主。

契约已确认后：`AF_DECIDE=ai` → **跳过**阶段内决策卡，直接落盘 + 结束闸门（`fast+ai` 且 A 档绿 → [免发卡](stage-delegation.md#fastai-免发卡审阅降频)；`strict+ai` → 审阅闸门卡）；`user` 或「这阶段我来」→ 问决策权/走澄清链（见 [stage-delegation.md](stage-delegation.md)）。

| 决策权 | 阶段内交互 | 阶段结束 |
|--------|------------|----------|
| **用户决策** | 按本文件下方规则（需求卡、技术栈等） | 阶段闸门 / 合并确认闸门 |
| **AI 自主** | **跳过**阶段内决策卡；Agent 推断并落盘 + `AI 决策记录` | **`fast+ai`+A档绿**：一行摘要+自动 skip_review；**`strict+ai`**：审阅闸门卡 |

用户说「这阶段你定 / 后面都你定 / 不审了继续」→ 见 stage-delegation 话术表。  
**提醒**：AI 自主只少问用户；**阶段顺序与 sol/dev 质量不变**（见上方铁律）。

## 对比总表

| 维度 | 快速 | 严谨 |
|------|------|------|
| **阶段顺序** | req →（按需）mod → sol → ①→②→可运行闸门→③；禁止跳阶段 | = 快速 |
| **sol 质量** | F 边界可对 AC；contracts 字段完整；architecture+详细 todo | **= 快速（同质）** |
| **dev 质量** | 按风险档：摘要五 bullet + 原子步骤表落点；禁止模糊句 | **= 快速（同质）** |
| **atlas/todo 详细度** | 流程进度 + 每 T 单独 ①②③；禁止摘要/合并多 T | = 快速 |
| **阶段 1（仅 user）** | 充分则跳澄清；否则缺口卡 → 写 REQ → **确认+闸门合并 1 卡** | 同左充分/缺口；写 REQ → 确认卡 → 闸门卡（分步） |
| **阶段 1–3（`AF_DECIDE=ai`）** | 落盘 → **`fast+ai`+A档绿**：一行摘要+自动 skip_review → 停（**无**审阅卡）；闸门未绿仍须审阅卡 | **`strict+ai`**：落盘 → **审阅闸门 1 张** → 停（多一张卡，**不加厚** sol） |
| **阶段 2 建模** | 三层：`entities/` + `conceptual/` + `physical/` | = 快速 |
| **阶段 3（仅 user）** | 技术栈 → **落盘** architecture+**详细** todo → 确认+继续合并卡 | 技术栈 → 落盘 → 方案确认 → 闸门 |
| **阶段 4 加速** | ≥2 无冲突 T → **推并行启动卡**；开闸后批量①≤3 + Subagent 并行② | 默认串行；用户显式「并行」才开 |
| **阶段 4 任务间（未开并行）** | 逐 T：①→②→可运行→③（禁批量写完所有①再码） | = 快速 |
| **阶段闸门** | AI自主：`fast+ai` 免发卡 / `strict+ai` 审阅卡；user_decide→阶段闸门；**均须停、禁同回复跨阶段**（建模跳过快路径除外） | = 快速（但 `strict+ai` 必发卡） |
| **集成覆盖率** | 出覆盖率报告即可（不卡阈值） | 变更模块 ≥80%（**建议目标，非闸门**；无脚本校验，由 Agent 按变更范围争取） |
| **首行声明** | 可简写：`📍 Agileflow 快速·阶段4·T-003` | 完整格式见 SKILL.md |

## 快速模式：AskQuestion 合并规则

### 阶段 1 — 确认+闸门合并

写完 REQ 草稿后，**一张卡**同时含：

```yaml
questions:
  - id: confirm_requirements
    prompt: "需求清单是否正确？（列出 REQ 摘要）"
    options: [确认 / 补充 / 大改]
  - id: continue_next
    prompt: "确认后是否继续下一阶段？"
    options: [是，继续（建模或方案） / 否，暂停]
```

- 选「确认」+「是，继续」→ 标 REQ 已确认 → **下条回复**进阶段 2/3 落盘  
- 选「否，暂停」→ 仅标已确认，不写下一阶段  
- **仍禁止**同回复写 model/solution

### 阶段 3 — 先落盘再确认（快速也适用）

**铁律：未写入 `architecture.md` + todo 前，禁止发「方案确认」。**  
**落盘质量与严谨同档**——禁止「先薄写再补」。

**快速路径**（仍须两停，可少一停）：

1. **AskQuestion 技术栈**（可单独；或与模式/决策权合并）→ 停  
2. **下条回复**写 features（若未写）+ `architecture.md` + `code-patterns` + `todo`（颗粒度 → [solution-core](solution-core.md)）  
3. **一张卡**：方案确认 + 是否继续（合并）→ 停  
4. 用户确认+继续 → **下条**进开发（本卡已含继续则**不再**单独阶段闸门）

**严谨路径**：技术栈卡 → 停 → 写 architecture+todo → 方案确认卡 → 停 → 阶段闸门 → 停。

**禁止**：合并卡里先「确认方案」再写 architecture（确认发生在落盘前 = 违规）。

## 快速模式：dev 构思（与严谨同质）

> 档位定义 → [04-development §风险分档](../phases/04-development.md#风险分档)。完整 = 标准 5 段 + 字面量严检。  
> **快速不减步骤、不减落点、不标「待补齐」代替内容。**

**❌** 空段凑数 · 假标题（纯文本无 `##`）· 用「快速」降档 · 模糊句（「实现登录逻辑」「按需调整」）。

## 建模产物（快速 / 严谨统一）

满足「须建模」时，按概念 / 逻辑 / 物理三层落盘：

```
atlas/model/
├── README.md                         # 索引 + 状态（根目录唯一入口）
├── conceptual/                       # 概念层
│   ├── entity-relations.md
│   └── domain-rules.md
├── entities/                         # 逻辑层
│   └── {EntityName}.md
└── physical/
    └── schema.md                     # 物理层（无持久化写 N/A）
```

**禁止**平铺在 `model/` 根下；**禁止**用 `model-overview.md` 单文件收敛。  
快速和严谨模式下 model 产物结构相同；区别只在阶段闸门（`fast+ai` 免发卡 / `strict+ai` 审阅卡）。

## REQ 信息充分例外（快速·严谨共用）

> 裁决 → [SKILL 信息充分少问](../SKILL.md#裁决表冲突时以此为准)。仅 `user_decide`；`ai` 本就不发澄清卡。

用户原话（或完整 PRD / `req:` 正文）**同时具备下列 4 项** → **跳过第 1 步需求澄清整卡**，本条直接写 REQ 草稿：

1. **目标用户**（或等价：公众 / 自己 / 家庭 / 内部…）
2. **平台形态**（Web / App / 微信小程序 / API-only…）
3. **≥1 个**可独立验收的核心功能（可拆成 REQ）
4. **MVP/本批范围边界**（先做哪些 / 不做哪些，或「本批都要」）

另可辅助触发（非必需）：用户明确「按此写 REQ / 不要问了」；或 `req:` 前缀且正文已含上列 4 项。

**仍须**：写 REQ（+UID 按需）→ **确认卡/合并闸门**（快速可合并；严谨可分步）→ 停。  
**禁止**：变更已确认 REQ 时用本例外绕过 [change-management](../phases/change-management.md)。

首行声明须含：`信息充分：跳过需求澄清卡（依据：{摘录}）`。

信息不完整 → **缺口卡**：只问未覆盖项（见 [requirement-askquestion](requirement-askquestion.md)）；缺口 = 0 等同充分。

## 「连续做」与少卡（快速专属）

| 说法 | 谁可用 | 意思 | 不是 |
|------|--------|------|------|
| **连续做 / 全部做 / 直接全开发** | 快速 **与** 严谨 | 任务之间可连着做完整 ①→②→**可运行闸门**→③，少停 | 跳过①；跳过可运行闸门；压缩模板；**自动**并行；Subagent 批量写码；薄写 sol/dev |
| **快速 + 连续做** | **仅快速 + user** | 阶段内可少发**构思审阅**卡 | 跳过结束闸门；跳过 F/MVP 阶段性确认；薄写 dev |
| **快速 + 要快/加速** | **仅快速** | 阶段 4 **推并行启动卡**（有可并发 T 时） | 用减文档换速度 |
| **AI自主 + 连续做** | `AF_DECIDE=ai` | F/MVP 阶段性确认默认跳过；`fast+ai` 结束免审阅卡；`strict+ai` 仍须审阅卡 | 跳过可运行闸门；跳过①；同回复跨阶段写盘；薄写 |

> **完整档（高风险）**：不适用少构思审阅；但仍可逐 T 连续做；并发时每 T 仍须完整档原子步骤表。  
> **严谨 + user**：逐 T 完整段、可连续；结束闸门不可省；默认不推并发。  
> **严谨 + AI**：须审阅闸门卡；不可套用 `fast+ai` 免发卡。

**绝不可少**：可运行闸门、构思闸门/写码闸门、并行启动卡（若并行）；sol/dev **同质颗粒度**；`user` 另不可少 F/MVP 阶段性确认与阶段结束闸门；`strict+ai` 另不可少阶段结束**审阅闸门卡**；`fast+ai` 另不可少**一行审阅摘要 + 自动 skip_review + 停**。

**铁律不变**：大阶段结束 → 必须**结束处理**（user_decide→阶段闸门；`strict+ai`→审阅卡；`fast+ai`+A档绿→免发卡自动确认）→ 本回复停止。  
**连续做 ≠ 跳阶段**：仍须 req→mod→sol→dev 按序；只是阶段内少卡、任务间可连续（建模跳过快路径除外）。
