# dev 阶段规则说明书

> 对应实现：`index.mjs`（聚合入口） + `steps.mjs`（步骤解析工具）。  
> 权威流程规范另见 `phases/04-development.md` 与 `templates/dev-quickstart.md`。

## 1. dev 文档到底要回答什么问题

每个 `atlas/dev/T-xxx-*.md` 不是「写码日志」，而是**写码前的构思契约 + 写码后的验收证据**。它必须让另一个人（或 30 天后的自己）能不看聊天记录就回答：

1. **本 T 是谁？** → `## 摘要` 里的 **本 T**；
2. **边界在哪？** → `## 摘要` 里的 **做 / 不做 / 上游**；
3. **怎么验收？** → `## 摘要` 里的 **AC**；
4. **具体改哪？** → `## 步骤` 里的代码落点（`Class.method` / `path/` / `func()`）；
5. **跑通没？** → `## 结果` 里的编译 + 启动/冒烟 + exit0/✅/PASS。

## 2. 每个检查规则是干什么的

### 2.1 结构类（先保证文档有骨架）

| 规则 | 报错时机 | 为什么要这样 |
|------|----------|--------------|
| `DEV-SEC-摘要` | 缺少 `## 摘要` | 没有摘要，读者无法 3 秒定位这个 T 在做什么。 |
| `DEV-SEC-步骤` | 缺少 `## 步骤` | 没有步骤，等于没有写码前的具体计划。 |
| `DEV-SEC-结果` | 缺少 `## 结果` | 没有结果，等于没有验收证据。 |
| `DEV-FAKE-标题` | 用纯文本「范围：」而不是 `## 摘要` | 假标题无法被脚本/人类稳定识别。 |
| `DEV-FAKE-过短` | 文档去空格后小于档位要求字数 | 防止用一句「实现登录」糊弄过去。 |

### 2.2 摘要类（边界 + 上游 + 验收）

| 规则 | 报错时机 | 为什么要这样 |
|------|----------|--------------|
| `DEV-SUMMARY` | 没有 `## 摘要` | 同上。 |
| `DEV-SUMMARY-定位` | 摘要里没写 **本 T** | 必须明确这个 T 对应哪个 F-xxx / REQ-xxx，避免「实现登录」这种含糊描述。 |
| `DEV-SUMMARY-结构` | 标准+ 缺少 **本T/做/不做/上游/AC** 五类 bullet | 五类 bullet 是防范围蔓延和验收扯皮的最小信息集。 |

### 2.3 步骤类（把「做」拆到可执行粒度）

| 规则 | 报错时机 | 为什么要这样 |
|------|----------|--------------|
| `DEV-STEP-最少` | 步骤数少于档位要求（精简≥1 / 标准≥2 / 完整≥3） | 防止「一步写完」导致思考粒度太粗。 |
| `DEV-STEP-FULL-须步骤表` | 完整档没用原子步骤表（`#### S1…` + 8 字段规格表）或流程表 | 完整档风险高，必须把每步的执行角色/触发条件/输入/处理逻辑(if-else)/调用依赖/异常处理(错误码+回滚)/输出/状态变更固定下来。 |
| `DEV-STEP-ATOM-字段` | 原子步骤表缺少必填字段（8 个：执行角色/触发条件/输入数据/处理逻辑/调用依赖/异常处理/输出数据/状态变更） | 每个步骤必须从 8 个维度完整描述，值可填"无"但行不能省——防止遗漏关键分支或异常处理。 |
| `DEV-STEP-ATOM-落点` | 原子步骤表「调用依赖」字段没有代码落点 | 调用依赖必须含 `Service.method(params)` 或明确填「无」。 |
| `DEV-STEP-流程落点` | 流程表「注意点」栏没有代码落点 | 步骤不能飘在天上，必须落在具体类/方法/路径上。 |
| `DEV-STEP-目的` | **5 列表头**下数据行未填「目的」列 | 旧 4 列（无目的）仍兼容；新 5 列须填目的。 |
| `DEV-STEP-FLOW-COLS` | 数据行与表头声明列数不一致 | 防止 5 列表头下数据行按 4 列写导致「目的/动作/注意点」静默错位。 |
| `DEV-STEP-改锚点` | **改 / 涉及改动** 行没有代码落点 | 步骤的改动必须可定位到文件或函数。 |
| `DEV-STEP-系统` | **系统** 行没有 AC-xxx / HTTP 状态 / toast 等可验信息 | 让步骤结果可对照 REQ 验收。 |
| `DEV-LIT-代码落点` | 全文没有 `Class.method` / `path/` / `func()` 反引号 | 防止整个文档都是业务描述，找不到具体代码位置。 |

### 2.4 链路与 SSOT 类（只链不抄）

| 规则 | 报错时机 | 为什么要这样 |
|------|----------|--------------|
| `DEV-LINK-API` | BE dev 没有链 API-xxx | 后端实现必须对应已确认的契约，不能私自改接口语义。 |
| `DEV-LINK-UI` | FE dev 没有链 UI-xxx / UID-xxx / F-xxx | 前端实现必须对应已确认的设计与边界。 |
| `DEV-LINK-UI-API` | FE dev 链了 API 但没链 UI-xxx | 调 API 时，字段绑定以 UI-xxx 为准，不能绕过 UI 契约直接看 API。 |
| `DEV-BAN-映射` | dev 内出现字段映射表（表头含「字段」+「映射/后端字段/接口字段」） | 字段映射是 contracts/UI 的 §字段绑定 的职责，dev 只链。 |
| `DEV-COPY-JSON` | dev 内粘贴大段 JSON 契约（含多个 `"key":` 键值对） | 契约 JSON 应该存在 contracts/API，dev 只链。 |
| `DEV-BAN-联调卡` | dev 内出现「联调卡」 | 联调是开发后的协作，不应该塞进单个 T 的 dev 文档。 |

### 2.5 旧形态与冗余类（保持 v9.11 统一格式）

| 规则 | 报错时机 | 为什么要这样 |
|------|----------|--------------|
| `DEV-BAN-范围` | 出现 `## 范围` | 范围信息应并入 `## 摘要` 的 **做/不做**。 |
| `DEV-BAN-异常` | 出现 `## 异常` | 异常处理应作为步骤里的 **系统** 行或引用 AC-xxx。 |
| `DEV-BAN-AC` | 出现 `## AC` | AC 应放在摘要 bullet 和结果映射表，不要单独成节。 |
| `DEV-BAN-旧标题` | 出现 `## 一、目标` 或 `## 五、可执行方案` | 统一用 `## 摘要 / ## 步骤 / ## 结果`。 |
| `DEV-BAN-构思章` | 出现 `## ① 构思` | 构思状态由 todo.md 的 checkbox 体现，不要重复成节。 |
| `DEV-BAN-关键实现点章` | 出现 `## ② 关键实现点` | 统一用 `## 步骤`。 |
| `DEV-LEGACY-段` | 出现 `## 前置 / ## 必读 / ## 契约` | 这些旧段名已收敛到 `摘要/步骤/结果` + 链 sol。 |

### 2.6 结果与验收类（证明它真的跑通了）

| 规则 | 报错时机 | 为什么要这样 |
|------|----------|--------------|
| `DEV-AC-MAP` | `## 结果` 里缺少 AC 映射表（表头须含 `AC` / `AC ID` 与 `unit/验证/方法/方式/test/人工/冒烟` 任一列） | 勾 ③ 前必须明确每个 AC 是通过单测、人工还是冒烟验证的。 |
| `DEV-AC-UNIT` | BE 的 AC 映射表没有 `test/unit` 路径 | 后端建议一个 AC 对应一个单元测试，形成可重复验证。 |

## 3. 步骤的三种写法怎么选

| 形态 | 适合场景 | 例子 |
|------|----------|------|
| **原子步骤表** `#### S1…` | **标准 / 完整档（推荐）**；需要把 if/else 分支、错误码、回滚、状态变更写全 | `#### S1：接收支付请求` 下跟 8 字段规格表 |
| 流程表 `S1…`（兼容旧） | 标准 / 完整档；步骤之间有明确输入输出 | `\| S1 \| 接收请求并校验 \| 收请求 \| body → 校验通过 \| 在 \`AuthService.validate\` 中调用 schema 校验 \|` |
| `####` 小节 | 精简档；单点改动 | `#### 新增登录接口` 下写 **用户 / 系统 / 改** 三行 |

### 原子步骤表（推荐格式）

每步是一个 `#### S1：步骤名称` 小节，下面跟 `| 字段 | 内容 |` 表，8 个字段必须全列：

```
#### S1：接收支付请求

| 字段 | 内容 |
|------|------|
| 执行角色 | 前端 → 后端 PaymentController |
| 触发条件 | 用户点击【立即支付】按钮 |
| 输入数据 | orderId(Long), payChannel(String, 枚举: ALIPAY/WECHAT) |
| 处理逻辑 | ① 校验 orderId → 否 → 抛 INVALID_PARAM(1001) ② if payChannel 不在枚举 → 抛 UNSUPPORTED_CHANNEL(1002) |
| 调用依赖 | `PaymentController.createPayment(PaymentRequest req)` |
| 异常处理 | INVALID_PARAM(1001) → Result.fail(1001, "参数错误")；不记录日志 |
| 输出数据 | 校验通过的 orderId, payChannel |
| 状态变更 | 无 |
```

| 字段 | 回答什么 | 常见错误 |
|------|----------|----------|
| **执行角色** | 谁执行？从前端到后端的哪一层？ | 只写"系统"不写具体角色 |
| **触发条件** | 什么情况下执行这步？ | 省略不写或写"步骤 N 完成"（可写但不够） |
| **输入数据** | 进来什么数据？带类型和约束 | 只写参数名不带类型 |
| **处理逻辑** | 具体怎么处理？**所有 if/else 分支都要写** | 用"系统处理"糊弄；只写正常路径不写异常分支 |
| **调用依赖** | 调了哪些服务/方法？`Service.method(params)` → 返回类型 | 只写方法名不带参数和返回值 |
| **异常处理** | 每个异常的错误码 + 回滚动作 | 只写"返回错误"不写错误码和回滚 |
| **输出数据** | 出去什么数据？ | 省略不写 |
| **状态变更** | 数据库/Redis/内存状态有什么变化？ | 省略不写或写"无"但其实有变更 |

### 流程表（兼容旧格式，5 列）

```
| 步骤 | 目的 | 动作 | 输入 → 输出 | 注意点（含落点） |
```

| 列 | 回答什么 | 写什么 | 常见错误 |
|----|----------|--------|----------|
| **步骤** | 这是第几步？ | `S1` `S2` …，按执行顺序编号 | 用文字「第一步」代替 `S1` |
| **目的** | **为什么**要做这步？ | 一句话解释这步在 T 里扮演什么角色（如"确认数据存在再改，防空指针"），不是重复动作 | 把动作抄进来（如"查数据库"←这是动作不是目的） |
| **动作** | 这步**做什么**？ | 动词开头，简述操作（如"按 id 查通知""校验归属"） | 写成需求描述（如"需要支持已读功能"） |
| **输入 → 输出** | 这步的**进出**是什么？ | 左边进来什么、右边出去什么，用 `→` 连接（如 `id → 实体或空`） | 只写输入不写输出，或写成长句 |
| **注意点（含落点）** | 具体在**哪个文件/方法**改？有什么坑？ | ① 代码落点（`` `Class.method` `` / `` `path/` `` / `` `func()` ``）② 关键决策或边界条件（如"非本人当 404 不泄露存在性"） | 只写业务说明不写代码位置；落点用单词反引号（如 `` `Service` `` ← 不合格） |

注意点列里的五种说法速查：

| 说法 | 什么时候用 |
|------|-----------|
| **继续走** | 本步不改已有代码，靠现有逻辑跑（如"继续走 `JwtFilter`"） |
| **在 X 上加** | 扩展现有类/文件（如"在 `TodoService` 上加 `complete()`"） |
| **直接用** | 零改动引用已有组件/工具（如"直接用 `Badge` 组件"） |
| **照 Y 那套** | 对齐邻居文件/同模块已有写法（如"照 `pages/weight/index.tsx`"） |
| **新写** | 无合适落点，须新建文件/类（加半句原因） |

**旧 4 列格式**（无目的列）仍兼容：表头只有 4 列时不会报 `DEV-STEP-目的`；但如果表头是 5 列而数据行按 4 列写，会报 `DEV-STEP-FLOW-COLS` error（防止内容列静默错位）。
