## 规则（构思先行）

> **构思给人看，不是给脚本凑形。** 闸门验段标题、步数、边界挂钩、逻辑块「怎么做」编号；  
> 人验「看不看得懂」。过闸是底线；写得让人/下一个 Agent 能照着码才是目的。  
> 每段解释 + few-shot → [dev-granularity](../templates/dev-granularity.md)。

### A. 文档格式规则（dev 文件本身怎么写）

| 标记 | 须做到 | 说明 |
|------|--------|------|
| 🔴 | **构思先行** | 合规 dev 落盘 + gate 绿 → 才可 Write 业务源码 |
| 🔴 | **构思须落盘** | 正文在 `atlas/dev/T-*.md`；聊天不算 |
| 🔴 | **一任务一文件** | 1 T = 1 文件；逐 T 勾选 ①②③ |

| 🟠 | **步骤可执行** | 主流程 3～8 + 边界 ≥2 + 实现说明（逻辑块怎么做编号 ≥2；见 [dev-granularity](../templates/dev-granularity.md)） |

### B. 流程与协作规则（怎么推进、怎么并行）

## 任务编排

阶段 3 落盘 todo + 功能依赖表；阶段 4：**TodoWrite 展开每 T ①②③** → [orchestrator 阶段 4](../templates/orchestrator.md#阶段-4每-t-一次派活--内部分-①②③钉死) 每 T 派 1 次 role-dev。

- **「全部开发」** = 展开清单 → **扫描可并发** → 连做（`ai` 有 FE+BE/无依赖 T 则自动并行，非强行串行）
- **并行权威** → [#并行阶段-4](#并行阶段-4)（此处不重复 FE+BE / 开批规则）

## 质量要求

> 每个 T 同一写法：`ai`/`user` 只改停点，不改厚度。

| 必达 | 说明 |
|------|------|
| **摘要** | `- **本 T** / **做** / **不做** / **上游** / **AC**` 五 bullet，冒号后有实质 |
| **步骤** | 主流程 3～8 + 边界 ≥2（挂第 N 步）+ 实现说明（逻辑块怎么做编号 ≥2） |
| **落点** | `Class.method` 或路径含 `/`；禁 `pom.xml` 假锚点 |
| **字面量** | `--gate dev-step1-literal` exit 0 才能勾① |
| **结果** | 可运行证据（编译+启/冒烟+exit0/✅/PASS） |
| **todo** | `### T-xxx` + ①②③ |

**只链不抄**：BE 链 API-xxx；FE 链 UI-xxx（+UID）；dev 禁映射表与 JSON 粘贴。形态细则 → [dev-granularity](../templates/dev-granularity.md) · exemplar。

> 「全部做」= 催进度，不跳闸门、不薄写。要加速 → [推并发](#并行阶段-4)。

## 闸门（唯一路径）

```
Read exemplar → 写 dev →【构思闸门+字面量】→ 勾① →【写码闸门】→ ② →【可运行闸门】→ ③
```

| 时机 | 检查 | 核验 |
|------|------|------|
| 勾①前 | 文件存在 + 主流程/边界/实现说明可执行 + 非事后补写 + 读范例 + `dev-step1-literal` | 脚本硬挡 |
| 写码前 | ①已勾 + 单任务 + 写法锚点 + 链 sol | 纪律自检 |
| 勾③前 | 编译 + 启/冒烟 + 证据写入 `## 结果` | 脚本硬挡 |

### 编码中调整（② 写码时发现 ① 需要改）

> **允许迭代，不死守原设计。** 判定与级（L0–L3）→ **[change-management §纠偏阶梯](change-management.md#纠偏阶梯全阶段)**（**唯一 SSOT**）。  
> L1 轻量反馈 → [§轻量反馈回路](change-management.md#轻量反馈回路)。本段只保留 **`## 设计调整` 格式示例**（L0 落盘用）。

**`## 设计调整` 格式**：
```markdown
## 设计调整（② 编码中发现）
- 原主流程第 2 步改为先调 YyyService.check() 再 load → 已更新实现说明
```

**禁止**：用「设计调整」塞入根本性方案变更以逃避回 ①（如整个做法重写却标「调整」）。发现即违规，须回 ①。

### 可运行闸门 — 勾 ③ / 父任务 ✅ 前

> **权威检查表**（编译/能启能调/冒烟/证据 + 「不过→」列）→ **[dev §可运行闸门](../templates/dev.md#可运行闸门每-t--每模块强制)**。本文件不再复述表格，避免双维护漂移。

### 与阶段 5 入场门禁的关系（合并验证）

> **本闸门 = 阶段 4 每 T 的可运行验证。** 阶段 5 入场门禁与其**合并验证**：③ 证据**可解析** → 增量；**不可解析** → 全量重验（判据 → [05 §③ 证据可解析](../phases/05-testing.md#③-证据可解析硬定义)）。

| 场景 | 阶段 5 入场门禁动作 |
|------|---------------------|
| ③ 证据**可解析**（各 T `## 结果` 含编译/启/冒烟证据） | 只验**增量**：本次变更涉及的模块集成冒烟；不重跑已验证端 |
| ③ 证据**不可解析** | 全量重验：编译→启动→冒烟 |

阶段 5 仍负责 **AC 验收归档 + 全量回归**，但入场门禁与阶段 4③ 合并。

## ①②③ 要点

### ①②③ 分别是什么

| 阶段 | 做什么 | 产出 |
|------|--------|------|
| **①** | 构思落盘 | `atlas/dev/T-xxx.md` 写出，过构思闸门 |
| **②** | 按 **实现说明** 写码 | 业务源码 + UT；FE 字段对齐 **UI-xxx §字段绑定** |
| **③** | 可运行 + AC 验收 | 先过 **可运行闸门**，再对照 REQ AC 写 test/ac；`## 结果` 回填证据；**回填 REQ 表「AC 测试方法/状态」**（禁仍「③ 后填」）；父任务 ✅ |

### 阶段 4 工作流

#### 0. 入口检查

- 若 sol 确认后用户手动改了代码（git hash 变化）→ AskQuestion「检测到方案确认后代码已有变更。是否更新方案？(是→回阶段3 / 否→基于当前代码继续)」→ 停。
- **大型项目分批（T ≥ 20）**：当前批次 ±5 个 T 展开①②③；已完成批次折叠为 `已完成: T-001~T-015 ✅`；未开始折叠为 `待做: T-021~T-050`；按 F-xxx 切片分批。

#### 1. TodoWrite 展开

🔴 **先 TodoWrite 展开**：为每个 T 建 **①②③ 三条**（强制）；「全部开发」时①条目须先全部出现为 pending。禁止一条 Todo 多 T；未展开禁止 Write 业务源码（见 [todo.md](../templates/todo.md#todowrite-强制展开)）。

#### 2. 单 T 串行流程（一次派活）

权威 → [orchestrator §阶段 4](../templates/orchestrator.md#阶段-4每-t-一次派活--内部分-①②③钉死) · [role-dev](../templates/role/role-dev.md)

```
取 T → 派 role-dev（一次 · 内 ①→②→③）→ 回报
→ 总控：literal→勾① → `--gate write-code`→勾② → 证据/todo→勾③ → 下一 T
→ 全部 T 齐 → dev-complete → 标「开发实现 ✅」
```

- 默认一次只推进 1 个 T；🔴 **该 T ② 开始前禁止为其他 T 勾①**（并行例外见 parallel）。
- 🔴 **禁止** 1～2 个 Subagent「实现全部 BE/FE」；🔴 **禁止跳过可运行闸门**。

#### 3. F-xxx / MVP 切片确认

任一 **F-xxx / MVP 切片** 相关 T 全 ✅ → 模块编译+冒烟（可运行闸门）→

- **`user`**：阶段性确认卡 → 停；
- **`AF_DECIDE=ai`**：默认继续下一 F（用户说「先给我看/演示」才发卡）→ 见 [contract §7.3](../templates/contract.md#73-fmvp-阶段性确认仅-userai-默认不发)。

用户要「给用户看」或选「先给我看」→ 存在端全量启动冒烟通过 + 给出启动步骤；等用户反馈。

#### 4. 结束条件

全部 T ✅ **且** `dev` 数=T 头 **且** ① completed **且** 可运行证据齐 → **总控**更新 README → `ai` 总控直接进入阶段 5 归档 / `user` 总控发阶段闸门→停。

### 轻量反馈回路（= 纠偏 L1）

> dev → model/sol 待回溯。**dev 里发现 model/sol 需要小改时，不要每次都走 5 步变更管理。** 先记笔，后批量回溯。  
> 权威 → [change-management §纠偏阶梯](change-management.md#纠偏阶梯全阶段) · [§轻量反馈回路](change-management.md#轻量反馈回路)（含对照表 + 冲突裁定：同时命中轻量+完整→按完整处理）

## 并行阶段-4

> **快 = 并发，不是薄文档。** 同一 F 的 FE+BE、无依赖 T → 应当并行。

| 模式 | 默认 |
|------|------|
| **`ai`** | 进阶段 4 **必扫描**；≥2 可并发 → **自动并行**（首行声明，不问卡） |
| **`user`** | ≥2 候选 → [并行卡](../templates/contract.md#74-并行启动卡仅-userai-不问) → 停 |

**谁可并行（须同时满足）**：无 `depends_on` 互阻 **或** 同 F 的 FE+BE 各 1 T；步骤落点路径无交集；同文件/同锁区 → 不可同批。同批建议 ≤3。

**并行批次前总控扫描（写文件集）**：

1. 每 T 从 dev ① 或 sol T 头提取「写文件集」（实现说明 【新写/改动】 路径）
2. 两 T 写文件集有交集 → **不可同批**
3. 有交集但可拆分 → 串行或改 dev ① 落点
4. 登记 [active-edits](../templates/active-edits.md)

**>3 个可并发 T 时分批**：

1. 按依赖图拓扑排序；**每批 ≤3** 个无互阻 T
2. **批次内**并行派 role-dev；**批次间**串行——上批全部 ③ ✅ 后再扫下一批
3. 跨批次须重新检查 `depends_on` 与 [active-edits](../templates/active-edits.md) 锁区

维护 [active-edits](../templates/active-edits.md)。

**禁止**：有可并发 FE+BE 却在 `ai` 下默认串行；「全部开发」当并行许可；并行时薄写 sol/dev。
