# 流程模式：快速 vs 严谨

> **权威定义**：AskQuestion 次数、dev 段数、建模粒度均以本文件为准。  
> 首行声明须标注 `模式：快速` 或 `模式：严谨`。

## 铁律：快速 ≠ 跳阶段 ≠ 薄 todo（最高优先级）

> **哪怕是快速模式**，也必须按顺序走完整交付链；`atlas/todo` 仍然详细。  
> **「用户不用管」= 决策权 AI 自主**（少问选型），**不是**快速模式的含义。

```
需求澄清 → 数据建模（按需；跳过须写判定）→ 方案设计（含详细 todo）
  → 开发①构思落盘 → ②写码 → ③验收（含可运行闸门）
```

| ❌ 快速模式禁止 | ✅ 快速模式允许 |
|----------------|----------------|
| 跳过 req/sol 直接写码 | 少问、合并确认卡（见下） |
| 跳过方案就进开发 | 建模按需跳过（须写「建模判定」） |
| 因「快速」把 todo 写成摘要/合并多 T/扁平 `- [ ] T-001` | **todo 与严谨同级**：`### T-xxx` + ①②③（含 dev 路径） |
| 把「快速」当成「用户不用管」 | 「用户不用管」→ 设 **AI 自主**（[stage-delegation](stage-delegation.md)） |
| 把「快速模式」当成豁免里的「快速通道」 | 快速通道仅微型改动；见 [00-intent-routing](../phases/00-intent-routing.md) |

**两维正交（勿混）**：

| 维度 | 控制什么 | **不**控制什么 |
|------|----------|----------------|
| **快速 / 严谨** | 文档厚度（dev 段、model 单文件）、AskQuestion **合并次数** | 阶段顺序、todo 是否详细、能否跳过①/可运行闸门 |
| **用户决策 / AI 自主** | 选型/澄清要不要问用户 | 阶段顺序、atlas 是否落盘、todo 详细度 |

## 风险分档（替代 T≥3 强制严谨）

> **流程重量与任务风险成正比，不与任务数量成正比。** 详见 [04-development §风险分档](../phases/04-development.md#风险分档替代-t3-强制严谨)。

| 档 | dev 文档 | 闸门 | 触发条件 |
|----|----------|------|----------|
| **精简档** | 3 段 | 1 道 | **全部**：单端 · 无 DB · 无权限/支付 · <100 行 · 无跨模块 |
| **标准档**（默认） | 5 段 | 2 道 | 未命中精简全部条件，也未命中完整任一条件 |
| **完整档** | **同标准 5 段** + 字面量严检 | 3 道+grep | **任一**：支付 · 权限 · DB schema · 多模块 · >100 行 · 用户说「严谨」 |

**档位在阶段 3 拆 todo 时标注**：`### T-xxx：[端] 描述 — F-001 [精简|标准|完整]`  
**校验器**：`validate-atlas` 扫档位标注；完整档须过字面量 grep，精简/标准档不要求。  
**升档不降档**：编码中发现风险升高 → 补齐缺失段+闸门；已写段不删。

## 决策委派（与快速/严谨正交）

每阶段入口：**仅当** `todo` 决策委派未设全局 AI自主 时，问决策权（见 [stage-delegation.md](stage-delegation.md)）。全局已 AI自主 → **跳过**决策权卡，直接落盘 + 审阅闸门。

| 决策权 | 阶段内交互 | 阶段结束 |
|--------|------------|----------|
| **用户决策** | 按本文件下方规则（需求卡、技术栈等） | 阶段闸门 / 合并确认闸门 |
| **AI 自主** | **跳过**阶段内决策卡；Agent 推断并落盘 + `AI 决策记录` | **审阅闸门**（用户可不审直接继续） |

用户说「这阶段你定 / 后面都你定 / 不审了继续」→ 见 stage-delegation 话术表。  
**提醒**：AI 自主只少问用户；**阶段顺序与 todo 详细度不变**（见上方铁律）。

## 对比总表

| 维度 | 快速 | 严谨 |
|------|------|------|
| **阶段顺序** | **同左**：req →（按需）mod → sol → ①→②→可运行闸门→③；禁止跳阶段 | **同左** |
| **atlas/todo 详细度** | **同左**：流程进度 + 每 T 单独 ①②③；禁止摘要/合并多 T | **同左** |
| **阶段 1 AskQuestion** | 需求卡 → 写 REQ → **确认+闸门合并 1 卡** | 需求卡 → 写 REQ → 确认卡 → 闸门卡（3 停） |
| **阶段 2 建模** | 可单文件 `model-overview.md` 替代五件套 | 必须五件套齐全 |
| **阶段 3 AskQuestion** | 技术栈 → **落盘** architecture+**详细** todo → 确认+继续合并卡 | 技术栈 → 落盘 → 方案确认 → 闸门 |
| **阶段 4 dev** | **按档位**：精简3 / 标准·完整5（完整=同段+严检；**无强制段数**） | 同左 |
| **阶段 4 任务间** | 逐 T：①→②→可运行→③（禁批量写完所有①再码） | 同左 |
| **阶段闸门** | AI自主→审阅闸门；user_decide→阶段闸门；**均须停、禁同回复跨阶段** | 同左 |
| **集成覆盖率** | 出覆盖率报告即可（不卡阈值） | 变更模块 ≥80% |
| **首行声明** | 可简写：`📍 Agileflow 快速·阶段4·T-003` | 完整格式见 SKILL.md |

## 快速模式：AskQuestion 合并规则

### 阶段 1 — 确认+闸门合并

写完 REQ 草稿后，**一张卡**同时含：

```yaml
questions:
  - id: confirm_requirements
    prompt: "需求清单是否正确？（列出 REQ 摘要）"
    options: [确认 / 补充 / 大改]
  - id: continue_next
    prompt: "确认后是否继续下一阶段？"
    options: [是，继续（建模或方案） / 否，暂停]
```

- 选「确认」+「是，继续」→ 标 REQ 已确认 → **下条回复**进阶段 2/3 落盘  
- 选「否，暂停」→ 仅标已确认，不写下一阶段  
- **仍禁止**同回复写 model/solution

### 阶段 3 — 先落盘再确认（快速也适用）

**铁律：未写入 `architecture.md` + todo 前，禁止发「方案确认」。**

**快速路径**（仍须两停，可少一停）：

1. **AskQuestion 技术栈**（可单独；或与模式/决策权合并）→ 停  
2. **下条回复**写 features（若未写）+ `architecture.md` + `code-patterns` + `todo`  
3. **一张卡**：方案确认 + 是否继续（合并）→ 停  
4. 用户确认+继续 → **下条**进开发（本卡已含继续则**不再**单独阶段闸门）

**严谨路径**：技术栈卡 → 停 → 写 architecture+todo → 方案确认卡 → 停 → 阶段闸门 → 停。

**禁止**：合并卡里先「确认方案」再写 architecture（确认发生在落盘前 = 违规）。

## 快速模式：dev 构思（按风险档位）

> 段数只看 `[精简|标准|完整]`。完整 = 标准 5 段 + 字面量严检（不额外加段）。

| 档 | 须有 |
|----|------|
| **精简** | `## 范围` · `## 做法` · `## 结果` |
| **标准·完整** | + `## 契约` · `## AC` |

**❌** 空段凑数 · 假标题（纯文本无 `##`）· 用「快速」降档。

---

## 快速模式：建模精简

满足「须建模」时，可用 **单文件** 替代五件套：

```
atlas/model/
├── README.md
└── model-overview.md   # 含：实体列表、关系摘要、关键规则、表结构摘要
```

`README.md` 标已确认后等同五件套已确认。  
**严谨模式禁止**用单文件替代。

---

## 快速模式：REQ 入口例外

满足**全部**时可 **跳过第 1 步需求卡片**，直接写 REQ 草稿：

1. 用户 `req:` 前缀 **或** 明确「按此写 REQ / 不要问了」
2. 已含：目标用户、平台、≥1 核心功能、MVP 范围
3. 非变更已确认 REQ

仍须：写 REQ → **确认+闸门合并卡**（快速）或分步确认（严谨）。

## 「连续做」与少卡（快速专属）

| 说法 | 谁可用 | 意思 | 不是 |
|------|--------|------|------|
| **连续做 / 全部做 / 直接全开发** | 快速 **与** 严谨 | 任务之间可连着做完整 ①→②→**可运行闸门**→③，少停 | 跳过①；跳过可运行闸门；压缩模板；并行；Subagent 批量写码 |
| **快速 + 连续做** | **仅快速** | 阶段内可少发**构思审阅**卡（唯一可少的卡） | 跳过结束闸门；跳过 F/MVP 阶段性确认卡 |

> **完整档（高风险）**：不适用少卡规则；但仍可逐 T 连续做。
> **严谨模式**：逐 T 完整段、可连续；结束闸门不可省。

**可少的卡（仅快速）**：可选「构思审阅」。  
**绝不可少**：F/MVP 阶段性确认、阶段结束闸门、并行启动卡、构思闸门/写码闸门/可运行闸门。

**铁律不变**：大阶段结束 → 必须**结束闸门**（user_decide→阶段闸门；ai_decide→审阅闸门）→ 本回复停止。  
**连续做 ≠ 跳阶段**：仍须 req→mod→sol→dev 按序；只是阶段内少卡、任务间可连续。

## 正误示例

**✅ 快速·阶段1**：用户长 PRD + `req:` → 直接写 REQ → 合并确认闸门卡 → 停  
**✅ 快速·阶段3**：技术栈卡 → 停 → 写 architecture+**详细** todo（每 T ①②③）→ 确认+继续合并卡 → 停  
**✅ 快速·dev**：按档位写段（精简3/标准·完整5）；完整档跑 `--gate dev-step1-literal` → ①→②→可运行→③
**✅ 完整档·「直接全开发」**：高风险走完整档（5 段+严检）；逐 T 连续，不可空壳/批量①
**✅ UID**：§2 区域表 + ASCII 布局线条图（缺图不可确认）

**❌ 快速就跳过需求/方案直接写码**  
**❌ 快速就把 todo 写成「实现 MVP」一条**  
**❌ 把「快速」当成「用户不用管」（应用 AI 自主）**  
**❌ 高风险任务仍自称精简档**
**❌ 空 `##` 标题无正文过构思闸门**  
**❌ 假标题：纯文本「范围/做法」无 `##` / 无 `####` 步骤**  
**❌ 旧标题 `## 一、目标` / `## 五、可执行方案` 未改仍勾①**  
**❌ FE 无 `### 布局` 线条图仍勾①**
**❌ 先写完 T-002～011 的 dev 再统一写码**  
**❌ 摘要版 / 合并多 T / 未落盘就确认方案**  
**❌ UID/FE 布局只有文字表、无线条图**
