# AgileFlow 核心思想

> 把想法拆成可执行阶段，落到可运行交付——**流程本身就是产品**。

---

## 产品是什么

AgileFlow 要解决的不是「让 AI 多写点代码」，而是：

**把想法拆解到实现落地的一套流程。**

流程即产品。每个阶段有固定规范：先把业务与设计说清楚，再构思，再写码，再验收。

防 AI 幻觉、不听指挥、假装完成，是后来加上的加固——**服从拆解，不反客为主**。

用户把决策权交给 AI 时，摩擦可以少，证据不能少。「你定 / 别问我」= 少澄清、少审阅卡；≠ 跳阶段、≠ 空跑勾选、≠ 先码后补。

---

## 主链

```
req → model(按需) → sol → dev(①②③) → test
```

brownfield 另有前置 `init`（盘点已有代码与业务），不进主链「想法拆解」本体。

| 阶段 | 一句话 |
|------|--------|
| **req** | 把「要什么」写成可独立验收的需求 |
| **model** | 把「业务世界里有什么」说清楚——没有则几乎无视 |
| **sol** | 定边界、暴露面、技术摆法，拆成带 ①②③ 的开发任务 |
| **dev** | 先盘可复用资源再构思；写法给人看得懂；写码；按 AC 证明做对了 |
| **test** | 证明整批可交，而不是单测绿就算完 |

**续跑**：`user` 模式在阶段闸门后等人说「继续」；断点可用「继续 agileflow」。  
**`AF_DECIDE=ai`**：闸门绿后须同会话连做到终点（阻塞式派 Subagent），禁止派完一批等人说「继续」。

---

## 各阶段

### req — 需求澄清

**解决什么**：交付什么业务能力；怎样才算验收通过。

- 需求是**验收的唯一权威**。AC 用 Given/When/Then 写在 REQ 里；下游只引用，不另造验收故事。
- **一功能一份 REQ**。MVP 是范围标签，不是把多功能力揉进一份总览。
- 有界面时结构与交互在 UID；链了 UID 就必须有文件，禁止空链标「已确认」。
- 信息够清楚就少问；没说清的只问缺口。问清楚是为了落盘，不是为了多聊。

**结束时应有**：已确认的 REQ（+ 按需 UID），人知道「做什么 / 怎么验收」。  
**不要变成**：技术方案、接口明细、开发任务清单，或 REQ 挂 UID 目录却空。

---

### model — 数据建模（按需）

**解决什么**：业务实体、关系、规则、必要时的持久化形状。

- **看项目，不看仪式**。复杂实体/状态机 → 加重；简单 CRUD、无新实体 → **可几乎无视**。
- 「几乎无视」≠ 静默跳过：须留**建模判定**（依据写清）。
- 只回答业务世界是什么，**不**定义 API、不拆开发任务。

**结束时应有**：可用 model，或白纸黑字的跳过判定。  
**不要变成**：每个小改动都强制全量建模，或跳过成本与全量一样重。

---

### sol — 方案设计

**解决什么**：系统做哪些功能、边界在哪、对外暴露什么、技术怎么摆、开发按什么顺序做。

- 方案是**从需求到任务的桥**：F 说边界与暴露面；contracts 承载接口/页面规格；架构全局一份；任务进 `todo`（每任务 ①②③）。
- **F 极简**：边界 + 暴露面；联调细节、字段映射、验收复述各有权威源，下游只链不抄。
- 技术栈先有来源（人定 / AI 记 / 仓库已有），再写架构。
- 任务默认串行；`AF_DECIDE=ai` 的正当加速 = **并发作业**，不是薄写 F/契约/todo。
- todo 里 ①②③ 此时仍应未勾选——清单是承诺形状，不是完成证明。

**结束时应有**：可执行的功能边界 + 契约（按需）+ 架构 + 带档位的开发任务清单。  
**不要变成**：抄 REQ、提前写源码与测试报告、提前勾满 ①②③，或因「少停」写成空壳。

---

### dev — 开发实现

**解决什么**：把方案里的每个任务真正做出来，跑得起来、对得上 AC，且人看得懂 AI 怎么做。

- 每任务固定三步：**① 构思 → ② 写码 → ③ 验 AC**。无合规构思，禁止写业务源码。
- **勾选 = 承诺已兑现**：勾① 须有 `atlas/dev/T-*.md`；勾③ 须「## 结果」里有可运行证据。
- **① 叙述五段式**（唯一质量线 = 完整）：摘要 + 主流程 + 边界 + 实现说明 + 结果。逻辑块「怎么做」须编号 ≥2；边界 ≥2 且挂「第 N 步」。接现有代码写清继续走 / 在…上加 / 照… / 新写。细则 → `skills/agileflow/templates/dev-granularity.md`。
- **可运行先于「写完」**：编译 / 能启 / 本任务冒烟——证据进构思「结果」；单测绿 ≠ 可交。
- 默认一次推一个任务；要快 → **并发**（`ai` 可推并行启动卡；`user` 须人显式要）。

**结束时应有**：每个 T 有可读构思、有代码、有可运行证据、AC 对得上。  
**不要变成**：先码后补构思、一行糊弄、无视库存造轮子、用 checkbox 全绿代替真的跑过。

---

### test — 测试验收

**解决什么**：整批功能能否交付；质量与回归是否站得住。

- 阶段 4 的 ③ 已按 AC 写测并跑过；阶段 5 **不重写测试**，负责入场、归档、回归。
- **入场是 ③ 的延续**：同会话复用已有证据，只补跨任务集成；跨会话或证据丢了才全量重验。
- 单测绿、分层点名跑通，都不能单独冒充「全量可交」。

**结束时应有**：验收报告齐、回归结论清楚（PASS / 阻断 / 失败）。  
**不要变成**：同会话无意义三重复跑，或缺测却硬标完成。

---

### init — 项目盘点（brownfield，主链外）

从零新建**不做** init；有存量才盘点。盘点是为了后续少幻觉、少撞命名，不是第四条主链阶段。

---

## 委托交付（`AF_DECIDE=ai`）

常见原话：「你定」「别问我」「全程你看着办」。未说清谁决策时仍须启动卡问人。

| 可以少 | 不能少 |
|--------|--------|
| 阶段内澄清卡、技术栈追问 | 主链顺序、atlas 按阶段落盘 |
| 阶段结束少停（A 档绿 → 一行摘要） | todo ①②③、构思文件、可运行证据、REQ AC 回填 |
| 等人说「继续」才开下一批 | 闸门 exit 0；阻塞派活同会话循环到交付 |

**易混**：「你定」≠ 已答启动卡；≠ 跳阶段 / 豁免捷径 / 先码后补。AI 自主少问时，形柱与跑柱必须更硬，否则必偷懒。

---

## 加固：形 / 令 / 跑

AI 容易幻觉、不听指挥、空跑冒充完成，于是加了三层加固——**概念对等，执行有硬有软**：

| 柱 | 管什么 | 靠什么 | 约束力 |
|----|--------|--------|--------|
| **形** | 按阶段写对；勾选对得上文件；假证据过不去 | `atlas/` + `validate-atlas`（`TODO-CHECK-*`、`REQ-UID-断链` 等） | **硬阻断** |
| **令** | 听不听指挥；阶段能不能往下走 | 首启启动卡；`user` 阶段停点；`ai` 少停但须自治连做 | 纪律 |
| **跑** | 真能编译 / 启动 / 冒烟 | 可运行闸门；证据写入 `## 结果`；脚本不替你跑 | **硬阻断** |

委托交付下令柱变软时，靠形柱/跑柱硬挡补上——否则「全让 AI 干」就会空跑勾选、假装完成。

## 配套机制

| 机制 | 作用 |
|------|------|
| **决策维（ai/user）** | 只控停点与是否推并发；不减 sol/dev 质量；不跳主链、不取消①或可运行闸门 |
| **质量线** | dev 唯一完整档：叙述五段式 + 逻辑块怎么做编号；`AF_TIER=full` 不减厚度 |
| **豁免** | 问答/微型/hotfix 须首行声明；灰区 AskQuestion |
| **纠偏 L0–L3** | 中途发现不对先判级再动作；禁止硬扛继续写码 |

**证据容器**：`atlas/`（requirements / model / solution / dev / tests 物理隔离）。业务源码写在工程正常位置。  
**下游只链不抄**：验收看 REQ AC，线框看 UID，接口看 contracts，边界看 F。

---

## 非目标

- 不是「纯防幻觉工具」——防失控服从拆解
- 不做豁免场景的完整流水线（微型/hotfix/纯问答走豁免）
- 脚本不替用户编译运行——只验落盘之形与勾选是否对得上证据
- 默认串行；`ai` 可推并发，但「全部开发」≠ 自动启 Subagent
- 不把「AI 自主」做成薄文档或跳证据的快速通道
